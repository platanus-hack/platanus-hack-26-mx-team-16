### 01-legal-ethics
_A code-enforced legal/ethics invariant (attestation gate, automatic-only-passive, public-ranking-only-passive, operative passive whitelist, rate-limit + identifiable UA) centralized in a dependency-free common/legal package plus a cross-cutting invariant test suite._

- **Produce:** backend/src/common/domain/legal/ package (constants.py: SCANNER_USER_AGENT, API_SCAN_RATE_LIMIT, worker rate constants); host.py: is_gov_hostname/is_sensitive_hostname/resolve_host_flags (HostFlags); levels.py: is_active(), default_visibility(), AUTOMATIC_ALLOWED_LEVELS predicates; passive_profile.py: GOV_PASSIVE_PROFILE contract + assert_within_passive_profile(); attestation_gate.py: enforce_attestation() + AttestationRequiredError(422)/AutomaticActiveScanError(500); robots.py: RobotsPolicy Protocol (contract; impl in 04); Cross-cutting invariant test suite backend/tests/legal/ (the load-bearing deliverable)
- **Consume:** ScanLevel and ScanVisibility enums + scans/sites table columns (level, visibility, authorized, authorized_at, requested_by, is_gov) from 06-data-model; Existing SaaS Redis RateLimiter + create_rate_limit_dependency factory (boilerplate, already in repo); SAQ job queue / CronJob (boilerplate, already in repo); common DomainError base + domain enums/constants (boilerplate)
- **Notas:** This feature is a transversal invariant, not a module with its own endpoints; its enforcement is physically implemented inside 04 (worker), 08 (scheduler/ranking), 12 (API), and 13 (frontend), while it OWNS the single source of truth (common/legal predicates/contracts) plus the invariant test suite that those siblings must pass. So it both depends on and is consumed by 02/04/08/12/13 — but only as frozen contracts (the Protocol, the GOV_PASSIVE_PROFILE allow-list, enforce_attestation signature, default_visibility, AUTOMATIC_ALLOWED_LEVELS), so all of them can be built in parallel against those frozen interfaces and integrated/tested later. The only true build-time blocker is 06-data-model, because levels.py/attestation_gate.py reference ScanLevel/ScanVisibility and the scans/sites columns; plan §2.3 explicitly offers an escape hatch (define those enums temporarily in legal and have 06 re-export them), which would let 01 and 06 proceed in parallel from a frozen enum contract and downgrade the hard dep to a paper agreement. Per plan §5 build order, 06 is first ('bloquea todo'), then common/legal, then 12/04/08/13. canStartDayOne is false strictly because of the 06 enum/column references; everything else is the existing SaaS boilerplate it deliberately reuses (Redis RateLimiter, SAQ cron, DomainError).

### 07-scoring

- **Notas:** Hard blocker is 06-data-model: scoring is defined entirely in terms of concrete scans columns (penalty_raw, coverage, agentic_status, status, overall_grade) and the findings shape (source/severity/confidence/dedupe_key). It reads and writes those columns, so the table/contract must exist or be frozen before scoring can be finished. The dependencies on 04 (web findings + coverage[]) and 05/03 (agentic findings + agentic_status states) are SOFT: the scoring function operates on a frozen input contract (a list of deduplicated findings with severity/confidence/source + the coverage/agentic_status fields), so it can be built and unit-tested in parallel against fixture findings while the scanners and agent team produce the real data. Scoring itself is pure determinism and never calls the LLM (the 05 narrative is a downstream consumer, not a dependency). The leaderboard order contract (overall_grade ASC, penalty_raw DESC) is owned here and consumed by 08/13, and the score+grade feed 09-reporting — those are downstream and can integrate against the frozen output schema, so scoring does NOT depend on them. canStartDayOne is false purely because of the 06-data-model schema dependency; if that schema is frozen on paper, scoring can otherwise be developed against fixtures with no live upstream.
