**Verified TRUE (no fix; confirms the plan's load-bearing premises):**
- F1/F3 "no new endpoint, no lazy fetch": the LIST path serializes the **full** per-doc array via the same `WorkflowCasePresenter`/`WorkflowDocumentPresenter` the create path uses, batch-loaded (`list_by_case_ids`, no N+1). Only FE mapping changes.
- `caseNoun(workflow, locale, count)` exists with exactly that 3-arg signature (`case-noun.ts:20`), already imported in `cases-view.tsx`.
- `updater.py:44-45` does `if self.name is not None: case.name = self.name` with no guard — the rename hole B1 targets is real.
- Per-file dispatch: a 20-file drop fires 20 sequential round-trips (`file-upload-button.tsx:83` awaits each) — relevant to F6 optimistic-row design (20 PENDING rows appear one at a time). `_ensure_case` is idempotent per file via `external_ref="upload:<file_id>"`.

## Open decisions (Vic must close — not derivable from code)

- **D1 — Fate of the (now-removed) technical/processing-jobs view.**
  Options: **(a)** drop the goal; rely on the per-case "Actividad" tab for run-level debugging *(reco — the page is already dead code; recreating a cross-case runs view is net-new work for low user value)*. **(b)** recreate a demoted "technical runs" route + wiring under an advanced nav group. Blocks N1 scope.
- **D2 — Expand gate: which cases show a chevron / expand.**
  Options: **(a)** always-present chevron, expands even at 0 docs → "Sin documentos" *(reco — consistent with §4.3's mandated empty state)*. **(b)** ≥1 doc (ported-source default `>1`). Must be written into **both** §4.3 and F3.
- **D3 — Must the name-editability invariant also guard the CREATE paths, and M2M?**
  Options: **(a)** guard JWT CREATE + UPDATE; **exempt M2M** (machine clients own naming) — state as a locked decision *(reco — closes the FE-reachable holes; documents the M2M carve-out so it isn't a silent gap)*. **(b)** guard all three including M2M (strict invariant). **(c)** UPDATE-only (status quo of the plan — leaves both CREATE holes open; not recommended). Also pick CREATE-with-name semantics: **422 reject** *(reco, simplest)* vs **ignore + auto-name**. And: the updater must reject only when the name **actually changes** (compare to `case.name`) so a no-op name echo on a status PUT doesn't 422.
- **D4 — Case-centric "bulk N files → N cases" path.**
  The backend already supports it for free (omit `workflowCaseId`; `_ensure_case` mints one idempotent case per file). §4.1 closes it off without recording it as a deliberate trade-off. Options: **(a)** keep excluded — force create+name each dossier *(reco — matches the dossier mental model; avoids 20 unnamed dossiers)*. **(b)** add a secondary "subir como casos nuevos" that dispatches without `workflowCaseId`. Pure UX call.
- **D5 — Per-case naming on a multi_doc_dossier workflow that also contains auto-named per_upload rows.**
  `caseNameEditable` is per-**workflow**, but auto-named `file · 4hex` cases can exist inside a case-centric workflow (any upload/source/email ingest dispatched without a `workflowCaseId`). Options: **(a)** on a dossier workflow, the toolbar/upload path **always** attaches a `workflowCaseId` (never the bare per_upload path); the guard then allows renaming those rows too *(reco — keep the invariant at the workflow level; `splitCaseName` already renders auto-named rows cleanly)*. **(b)** add per-case name-editability. State the chosen invariant so F4 never renders a case-less `FileUploadButton` on a dossier workflow.
- **D6 — Empty 0-doc dossier lifecycle.**
  An empty user-named dossier is the expected initial state, but the plan gives no positive evidence that `EvaluateCaseCompleteness` / mark-ready / finalize tolerate a 0-doc case. Options: **(a)** add an explicit test/note that a fresh empty case renders + expands to "Sin documentos", cannot be marked ready/finalized prematurely, and completeness returns "incomplete" (not error) on zero docs *(reco)*. **(b)** leave to runtime discovery (not recommended).
