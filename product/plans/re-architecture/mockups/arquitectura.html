§0 La idea en una frase

Antes, cada flujo de procesamiento era código a medida: un workflow de Temporal distinto y hardcodeado para cada cosa (subir docs, ingerir de una fuente, re-extraer…). Cambiar el comportamiento significaba tocar y desplegar código.

Ahora hay un solo motor que no sabe nada de casos concretos: lee una receta (una lista de pasos guardada como datos) y la ejecuta. Cambiar el comportamiento es cambiar la receta — incluso desde una UI, sin tocar código.

ANTES — un motor por flujo

lógica hardcodeada · desplegar para cambiar

AHORA — motor único + recetas

pipeline-as-data · cambiar = editar datos

Por qué importa El flujo legacy no desapareció: quedó congelado como una receta más (standard-extraction@v1), con tests golden que garantizan que da byte-idéntico el mismo resultado. La migración fue un cambio de motor, no un cambio de comportamiento.

§1 Los tres conceptos núcleo

Todo el sistema se entiende con tres piezas. Si solo te llevas algo de este documento, que sea esto.

El que ejecuta

El Motor

PipelineInterpreter es un intérprete genérico sobre Temporal. No conoce "farmacias" ni "circulares": recibe una receta y va corriendo sus pasos uno a uno, guardando lo que cada paso produce.

El qué ejecutar

La Receta

Un PipelineVersion: una lista ordenada de fases, guardada como datos e inmutable (versionada). Cada workflow apunta a su receta por pipeline_id. Editarla crea una versión nueva; las anteriores siguen reproducibles.

Los pasos

Las Fases

Cada paso de la receta es una fase de un kind conocido (extract, analyze, human_review…). Son piezas intercambiables con su propio config. Armar un flujo = ordenar fases.

Analogía El motor es la cocina; la receta es la tarjeta de recetas; las fases son los pasos ("picar", "hervir", "emplatar"). Cambias el plato cambiando la tarjeta, no rehaciendo la cocina.

§2 Las fases — el vocabulario del sistema

Una receta encadena fases. Cada fase tiene un scope: corre por documento (una vez por cada archivo) o sobre el caso completo (el expediente, y puede pausar de forma durable). Un flujo típico de expediente se ve así:

classify_pages→ extract→ assess→ enrich→ await_documents→ analyze→ confidence_gate→ human_review→ output→ deliver
scope documento scope caso decisión / pausa salida
FaseScopeQué hace, en simple
classify_pagesdocumentoMira las páginas y decide qué tipo de documento es cada una. Opcionalmente parte un PDF multi-doc en varios casos hijos (ver §7).
extractdocumentoSaca texto y campos estructurados. El extractor se elige por tipo: OCR, VLM (manuscritos, vía Gemini Vision) o ASR (audio→texto). auto despacha solo.
assessdocumentoUn agente LLM evalúa qué tan seguro está de cada campo (confianza capa-2), deja señales cualitativas y hasta 3 candidatos. "¿Confío en lo que leí?"
enrichdocumentoLlama a un servicio externo del cliente (HTTP firmado) para traer datos extra. El resultado se guarda como un documento TOOL (ver §3).
await_documentscasoEspera durable: ¿el expediente está completo según la CompletenessPolicy? Aguanta hasta que llegue todo lo requerido o alguien marque "listo".
analyzecasoCorre las reglas de negocio sobre los datos del expediente (es un child workflow de análisis). Produce veredictos PASS/FAIL por regla.
confidence_gatecasoEvalúa la ActivationPolicy: ¿la confianza alcanza?, ¿hay reglas bloqueantes? Decide entre seguir, pedir aclaración o mandar a revisión.
human_reviewcasoPausa durable para revisión humana por niveles L1 (analistas Doxiq) → L2 (cliente), con verificación por campo, lock y correcciones (ver §6).
await_clarificationcasoEmite una solicitud de aclaración estructurada al cliente y espera. El loop lo gobierna el cliente; Doxiq se mantiene stateless.
outputcasoProyecta los datos + resultados de reglas a un JSON de salida limpio (con citations y referencias @rule.<slug>).
delivercasoEntrega ese output a los destinos configurados vía webhook firmado (HMAC estilo Svix).
El truco de los scopes Las fases de documento corren en paralelo por cada archivo; las de caso esperan a tener el expediente entero. El motor sabe cuándo "subir" del documento al caso, y las fases de caso pueden dormir días (esperando un doc o una firma humana) sin consumir recursos — eso es lo que aporta Temporal.

§3 El concepto unificador: documentos virtuales

Problema: no todo lo que entra a un expediente es un archivo escaneado. A veces es datos que manda el cliente por API, o la respuesta de una herramienta externa. ¿Cómo hacemos que las reglas funcionen igual sobre todo eso?

Solución elegante: todo es un WorkflowDocument. Lo que cambia es su origen:

source = UPLOAD / SOURCE

Documento real

Un PDF, imagen o audio que llegó por upload o por un canal. Se le hace OCR/VLM/ASR.

source = EXTERNAL_DATA

Documento virtual

Datos JSON inyectados por el cliente vía POST /v1/cases/{id}/data. No hay archivo; el payload es la extracción.

source = TOOL

Resultado de tool

Lo que devolvió una fase enrich al llamar un servicio externo. Se guarda como un doc más.

Por qué es clave Como los tres son WorkflowDocument, una regla como @persona.nombre funciona sin cambios sin importar si el dato vino de un escaneo, de la API del cliente o de un servicio externo. Una sola abstracción elimina tres casos especiales. Por dentro, el motor lleva un PipelineState = un mapa de "artefactos" que cada fase lee y escribe.

§4 El Caso y su ciclo de vida

Un Caso (expediente) es la unidad de trabajo end-to-end: agrupa todos los documentos de un mismo trámite y lleva su propio estado. Hay una máquina de estados pública de 11 estados con transiciones legales — nada de cambiar de estado a mano por fuera. Todo pasa por un único choke point (TransitionCaseStatus) que además registra cada cambio en case_events con clave de deduplicación para reintentos.

RECEIVING→ PROCESSING→ NEEDS_CLARIFICATION↺ ANALYZING→ NEEDS_REVIEW→ REVIEW_L1→ REVIEW_L2→ COMPLETED ·REJECTED ·FAILED
  • Completitud explícita — un expediente multi-doc no avanza hasta tener todo lo requerido (CompletenessPolicy), o hasta que alguien lo marque ready (con confirmación force si falta algo).
  • Durabilidad — el caso vive en un workflow durable CASE#… que puede pausar en revisión o esperando docs, y reanudar exactamente donde quedó vía señales de Temporal.
  • Trazabilidad — cada transición queda en el timeline; el output sella la versión de pipeline y de cada tipo de documento que se usó (reproducibilidad total).

§5 La capa de decisión: policies declarativas

¿Quién decide si un caso pasa solo, necesita aclaración o va a revisión humana? No es código if/else regado: son dos policies declarativas que viven dentro del PipelineVersion (inmutables, versionadas).

¿Está completo?

CompletenessPolicy

Define qué documentos/tipos hacen falta para considerar el expediente completo. La fase await_documents la evalúa en loop hasta cumplirla (o hasta un ready manual).

¿Pasa, aclara o revisa?

ActivationPolicy

Combina umbrales de confianza (por campo → por tipo → default), reglas bloqueantes y muestreo. La fase confidence_gate la evalúa y elige el camino: seguir, gate.clarify o gate.review.

Dos tipos de muestreo La ActivationPolicy tiene sample_rate ("manda un % a revisión aunque pase") y qa_sample_rate ("audita un % después de completado"). El primero frena el caso; el segundo crea una tarea de QA para staff sin tocar el run, que ya cerró.

§6 Revisión humana multinivel (L1 / L2)

Cuando un caso necesita ojos humanos, la fase human_review lo pausa y abre tareas en stages:

Stage L1

Analistas de Doxiq

Una consola staff cross-tenant de Llamitai. Un analista puede ver casos de cualquier tenant — pero solo los que estén ligados a una tarea L1 que le tocó (sino, fuga de PII). Todo queda en un log de auditoría append-only.

Stage L2

El cliente

El equipo del cliente aprueba/rechaza en su propio bench, viendo solo lo que L1 no resolvió ya (filtro por nivel de verificación). Rechazo en cualquier stage ⇒ el caso termina REJECTED, sin output ni deliver.

  • Verificación por campo — cada campo lleva {value, verified_by, level, …}. Corregir un valor re-dispara el análisis (señal corrections) antes de permitir aprobar.
  • Claim / lock — un revisor "reclama" una tarea; si otro intenta editar campos de una tarea reclamada, recibe 423 case.locked. Las resoluciones son updates condicionales (a prueba de carreras).
  • Permisos por acción — el código chequea la acción (operate, approve, manage), nunca el nombre del rol. La matriz rol×acción es por workflow.

§7 Fan-out a casos hijos

Algunos documentos son en realidad varios: un PDF de circulares judiciales con una portada + 3 oficios distintos. La fase classify_pages con fan_out los parte en casos hijos independientes:

  • La portada se queda en el padre; solo los tipos listados en fan_out_types se separan en hijos.
  • Cada hijo recibe una clave determinista namespaceada por origen — un segundo archivo crea hermanos nuevos, no pisa los existentes; y nunca "adopta" casos ajenos.
  • Cada hijo corre su propio ciclo (extracción, reglas, revisión). El padre queda en PROCESSING y auto-COMPLETA cuando todos los hijos terminan.
Esto era imposible antes El motor legacy no tenía noción de "un trabajo que genera más trabajos". El fan-out a child cases es justo lo que desbloqueó el Caso 3 (circulares judiciales).

§8 Reglas y tokens — el lenguaje de negocio

Las reglas de negocio se escriben con un mini-lenguaje de tokens. La regla mnemónica: @ mira adentro (datos del expediente), # mira afuera (conocimiento y servicios externos), {{}} es runtime:

@slug.pathDatos del caso: un documento por su slug, un campo por su path. P.ej. @poliza.numero. Las llaves son opcionales.
@rule.<slug>El resultado de otra regla. (rule es un slug reservado.) Permite componer reglas.
#kb.<slug>Una referencia a la base de conocimiento del workflow. (La forma plana #slug sigue funcionando como alias.)
#tool.<slug>.pathLa respuesta de una tool HTTP firmada, resuelta lazy: el servicio se llama solo si una regla activa lo referencia; la respuesta queda auditada como documento TOOL. P.ej. #tool.dian.estado == "ACTIVO". PROPUESTO · spec rule-tools.md
{{system_var}}Variables de runtime que resuelve el sistema, como {{now}}.
  • when por regla — una regla puede aplicar solo si un predicado es verdadero: when @persona.tipo_entidad == "natural". Si el predicado es falso, la regla queda SKIPPED y no afecta el veredicto ni la confianza.
  • Checksums multi-país — validadores integrados de identificadores: NIT/CI Bolivia, NIT/CC Colombia, RUT Chile, Luhn, IBAN…
  • Slugs estables — los slugs de reglas y tipos de documento sobreviven a los renames; export/import los preserva.

§9 Conectividad: cómo entran y salen los datos

Entrada · canales nativos

Channels provider-agnostic

Un contrato único ChannelMessage con adapters para email (Mailgun/Mailpit) y WhatsApp Cloud. Cada proveedor verifica su firma (Meta X-Hub-Signature-256, Mailgun HMAC). Dedup por Message-ID/wamid. El audio entra solo por canal y va al extractor asr.

Entrada · API M2M

POST /v1/cases

API plana para integraciones: crear caso (find-or-create por external_ref), inyectar datos (/data), leer estado y output. Pensada para que el cliente orqueste sin saber de Temporal.

Salida · enrich

Tools HTTP firmadas

La fase enrich llama servicios del cliente con firma HMAC verificable offline, valida entrada/salida con JSON Schema y maneja fallos (review/continue/fail). Es un diferenciador: ningún IDP del mercado lo ofrece nativo.

Salida · deliver

Webhooks firmados

El output se entrega a destinos opt-in por workflow con webhooks firmados. Eventos del ciclo: case.created, case.needs_review, case.review.completed, case.output.ready, case.failed.

§10 La plataforma se vuelve self-service (E6)

El gran salto final: dejar de configurar workflows por código y dárselo al usuario.

Editor visual

/pipelines editable

Un stepper de tarjetas de fase (drag & drop) alimentado por un catálogo único de fases. Validación continua con el mismo 422 que el publish, modal de diff estructural e historial de versiones.

Portabilidad

Export / import bundle

Exportar un workflow completo (tipos de doc + reglas + pipeline + policies) a un JSON git-able, sin secretos. Importar en otra cuenta con find-or-create por slug. Hace los workflows versionables y compartibles.

Arranque rápido

Plantillas

Plantillas estáticas ("crear desde plantilla") para que un cliente nuevo tenga un workflow funcional en segundos.

Calidad continua

QA sampling

Tras completar, un % de casos auto-aprobados se sortea (determinista) para auditoría de staff. Métricas en /staff/v1/metrics.

§11 Cómo fluye un documento — ejemplo end-to-end

Ejemplo concreto: un pedido de helados que llega por email con un PDF (Caso 4).

Llega por canal

El email entra por POST /v1/channels/email/…. Se verifica la firma del proveedor y se deduplica por Message-ID. El adjunto se baja en background tras responder 200.

Se crea el caso y arranca el motor

Se crea un WorkflowDocument y se dispara el PipelineInterpreter con la receta del workflow "Pedidos Heladería".

Extracción automática

El extractor auto ve que es un PDF → OCR. (Si hubiera sido un audio adjunto, habría ido a asr → Gemini transcribe el pedido.)

Reglas de negocio

La fase analyze corre las reglas: por ejemplo, @pedido.total > 0 ⇒ PASS.

El gate decide

confidence_gate evalúa la ActivationPolicy. Confianza alta y sin reglas bloqueantes ⇒ straight-through, sin revisión humana.

Salida y entrega

output arma el JSON limpio del pedido; deliver lo manda al ERP del cliente con un webhook firmado (case.output.ready). El caso queda COMPLETED.

Lo importante Ese mismo motor, con otra receta y otras policies, atiende farmacias por WhatsApp, fondos de garantía bancarios con doble aprobación y circulares judiciales con fan-out. Una sola pieza de código, N comportamientos — definidos como datos.
