00 La idea

Las dos puntas del pipeline, configurables

Un pipeline tiene una entrada (ingest) y una salida (finalize). Este plan las vuelve configurables por workflow mediante Conexiones: Sources alimentan la entrada, Destinations consumen la salida (en la UI del producto: «Orígenes/Destinos»). HTTP webhooks primero (ver §08).

→ Entrada · Sources
  • Webhook — estilo n8n, protegido por API key o HMAC. 1ª entrega
  • Drive — una carpeta; cada archivo subido entra al workflow.
  • Email — un alias; cada correo carga su body + attachments.
  • WhatsApp — un número (vía kapso, proveedor del API de WhatsApp que ya usa el producto); los archivos enviados entran.
  • Slack — solo salida: su valor aquí es notificar resultados, no ingerir archivos (Destination en F12).
← Salida · Destinations
  • Webhook HTTP — uno o más, cada uno con su secret (firma Svix-compatible + reintentos). 1ª entrega
  • Drive — deposita los outputs como JSON en una carpeta.
  • Slack — notifica al terminar el procesamiento.

El payload deriva del output del pipeline (W1, ver §06); el destination solo elige a qué eventos se suscribe.

01 Encaje con el pipeline

Sources → ingest … finalize → Destinations

Las conexiones son el front y el back configurables del pipeline. No cambian el motor: el ingest se alimenta de un Source resuelto; el finalize entrega a los Destinations suscritos.

◆ Entrada de archivos vs datos estructurados (W2)
Toda ingesta de archivos pasa por un Source (el upload HTTP es el source webhook). La entrada de datos estructurados — farmacia B (POST /case con meds validados) y los callbacks de HumanTask (POST /review, /resolve) — no es un Source: es una API de control aparte, porque no carga archivos.
02 Sources (entrada)

Recibir archivos y dispararlos a un workflow

Un Source resuelve una identidad ruteable (token / alias / número / carpeta) → workflow_source → workflow → pipeline. La primera entrega activa solo WEBHOOK.

POST /v1/ingest/{token} · multipart
# el token (col dedicada, D7) resuelve: token → workflow_source → workflow → pipeline
POST /v1/ingest/src_7f3c...        # token público ruteable (prefijo src_)
X-Api-Key: dxk_...                 # auth_mode=api_key (default, D6); o firma HMAC
Content-Type: multipart/form-data
{ file: <bytes> }
  → 202 { jobId, status: "processing" }     # descarga a S3 + dispara el pipeline Temporal

03 Destinations (salida)

Enviar el resultado a donde el workflow lo necesite

Un Destination se suscribe a eventos del catálogo; al dispararse, el dispatcher arma el payload (W1, ver §06) y entrega con firma + reintentos.

POST {url} · webhook de salida (firma Svix-compatible)
// evento analysis_run.completed — payload = output_schema del pipeline
POST https://cliente.com/hook
User-Agent: Doxiq-Webhooks/1.0
Doxiq-Id: evt_…
Doxiq-Timestamp: 1765900000
Doxiq-Signature: v1,<base64(HMAC-SHA256)>   # HMAC con el secret whsec_… del destination
{
  "event": "analysis_run.completed",
  "workflowId": "wf_...", "caseId": "case_...",
  "output": { … }              # conforme al output_schema del pipeline
}

Contenido firmado = "{event_id}.{timestamp}.{body}" (Svix-compatible) · secreto whsec_… · User-Agent: Doxiq-Webhooks/1.0 — fuente: signing.py.

04 Inline vs ConnectionAccount

El webhook va inline; el OAuth va al registro org

La pregunta de fondo: ¿registro org-level (ConnectionAccount) o config per-workflow? La respuesta sale de la naturaleza de la credencial de cada provider — es exactamente el modelo de n8n (Credentials reusables vs parámetros del nodo).

⚠ Este plan supersede parte de specs/connections
product/specs/connections/spec.md §4.4 planeaba migrar el webhook a un ConnectionAccount tipo webhook. Se invierte (D2): el webhook queda inline (account_id=NULL); ConnectionAccount se reserva para los providers OAuth. El provider WEBHOOK del enum queda candidato a eliminarse.

El enum ConnectionProvider incluye además HTTP (capability LOOKUP, para Tools — ver F5); no es un Source ni un Destination.

◆ Por qué catálogo chico (W1)
Los docs de casos inventaban coverage.completed / circular.completed / resolution.completed. Ahora todos colapsan en analysis_run.completed y lo que cambia es el output_schema del payload. El enum queda estable y fiel al código (common/domain/enums/webhooks.py); la "salida configurable" es el output_schema, no el nombre del evento.
07 Auth de sources

API key (default) y HMAC · prefijos unificados

Auth del source (D3 · D6)

  • API key (default, D6): X-Api-Key: dxk_… verificado contra hash.
  • HMAC: firma del body + timestamp anti-replay con hmac_secret.
  • Cada source declara su auth_mode (api_key | hmac | both); default api_key.

Esquema de prefijos

  • src_ — token público ruteable del source (en columna, D7).
  • dxk_ — API key del source (verificable contra hash).
  • whsec_ — secreto HMAC: firma saliente del destination y verificación entrante del source.
✔ Groundwork reusable (F0)
Un value-object "secreto revelable + rotación" unifica el patrón repetido (webhook_destination.secret, tenant.webhook_signature_key, los nuevos api_key/hmac_secret del source): generar con prefijo, String(512), reveal/regenerate, presenter hasX. Reusa SSRF/URL-validation, firma+POST+retry y el pipeline de ingesta (S3 + Temporal) ya existentes.
