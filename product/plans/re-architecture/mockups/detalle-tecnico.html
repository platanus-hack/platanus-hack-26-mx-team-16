Cómo queda todo con el refactor

Modelos, endpoints, flujo y UI — concreto y anclado al código actual. La buena noticia primero: el modelo de hoy ya tiene casi todo el andamiaje. El refactor es mayormente aditivo, no una reescritura.

◆ Estado del código (corte 2026-06-09 · rama feat/re-arch, working tree)
Gran parte de este diseño ya está implementada y sin commitear: motor de pipelines (PipelineInterpreterWorkflow, PHASE_LIBRARY con 11 de 13 fases), HumanTask, Tools (tool_definitions), Sources (workflow_sources), workflow_destinations y TenantApiKey. Las secciones "hoy / actual" describen el estado pre-branch salvo nota explícita. Pendiente: handlers evaluate_rules/synthesize, seed del pipeline standard-extraction, API de autoría de pipelines, y migrar el dispatch de UI al intérprete.

02 Endpoints

La API interna sigue; se agrega una API pública M2M

Hoy todo pasa por la API interna (JWT Bearer + header X-Tenant + verify_workflow_access) que usa la UI, con una secuencia de 4–5 llamadas para procesar un caso. El cliente B2B no debe orquestar eso: se le expone una API pública machine-to-machine (M2M) con X-Api-Key tenant-scoped donde A y B son una sola llamada cada uno.

Los ejemplos usan el caso Farmacia (Procesamiento y validación de recetas médicas) como conductor; Garantías y Circulares usan los mismos modelos y endpoints.

Hoy: la UI orquesta 4–5 llamadas

Registro de eventos: webhooks vs señales internas

El catálogo es un enum chico (W1): el payload de resultado lo determina el output_schema del pipeline, no un evento por caso. La salida (Destinations: webhook/Drive/Slack) y la entrada (Sources) están en el plan de Conexiones. Para no confundir: los webhooks salen al cliente — incluidos los de Human-in-the-Loop (HITL); las señales son internas de Temporal y nunca se exponen.

04 Tareas humanas y pausas

HumanTask — una primitiva para todas las pausas

Las fases nuevas que detienen un pipeline son de dos familias, y todas usan la misma primitiva de Temporal (wait_condition + signal). Las que esperan a una persona crean un HumanTask unificado — así la "aclaración" de farmacia y la "aprobación" del banco son la misma entidad con distinto kind.

Dos familias de pausa

FaseEspera a…¿HumanTask?Quién resuelveReanuda con
await_documentsque lleguen los doc types requeridosNo · system-waitel sistema (evento de upload)document.uploaded → re-evalúa completeness
await_clarificationque un humano aclare campos de baja confianzaSí · kind=clarificationcliente (WhatsApp) o cola internatask_resolved
human_reviewque un humano apruebe/rechace el resultadoSí · kind=approvalcliente (callback) o analista internotask_resolved

Farmacia usa await_clarification (cliente, WhatsApp). Banco usa human_review (cliente, callback); Garantías NO usa await_documents — procesa incrementalmente con on_empty=SKIPPED. await_documents queda como capacidad de la plataforma (espera datos, no personas) para casos que necesiten barrera de completitud.

La entidad unificada

NUEVO HumanTask  ya en feat/re-arch

Unifica los dos conceptos previos del diseño (solicitud de aclaración y tarea de revisión) en una sola entidad.

HumanTask
  id, tenant_id
  case_id              # expediente (= WorkflowCase en el dominio bancario)
  pipeline_run_id      # qué run pausar/reanudar (target del signal)
  kind                 # clarification | approval
  status               # pending | resolved | rejected | expired | canceled
  assignee_mode        # internal_queue | external_callback
  audience             # doxiq_analyst | bank_analyst | … (rol/cola; distingue reviews encadenadas)
  payload              # qué se pide (campos) o se muestra (borrador + rule_results)
  resolution           # {fields:{...}} | {decision, reason}
  resolved_by          # user interno | "callback:tenant"
  created_at, expires_at, resolved_at

¿Por qué una sola entidad?

La aclaración (farmacia) y la aprobación (banco) son la misma forma: una tarea que pausa un run hasta que un humano responde. Difieren solo en kind y en la forma de payload/resolution.

  • Una tabla, una cola, una lógica de expiración y un mismo endpoint de resolución.
  • Reusa el mecanismo de pausa durable (no hay 2 implementaciones).
  • Extensible: kind=manual_split, kind=exception… sin entidades nuevas.

Las 3 fases en config

# 1) espera de datos — sin HumanTask (capacidad; garantías la omite, va incremental)
- id: await_documents
  params: { require_types: ["anexo","evaluacion","resolucion"], on_timeout: "expire_after_7d" }

# 2) aclaración (farmacia) — HumanTask kind=clarification
- id: await_clarification
  params: { trigger: "low_confidence_fields", assignee: "external_callback", emits: "needs_clarification" }

# 3) aprobación (banco) — HumanTask kind=approval
- id: human_review
  params: { assignee: "external_callback", emits: "review.pending", on_reject: "return_to_rules", expires: "7d" }
◆ Interno vs externo
assignee_mode decide quién resuelve: internal_queue (un analista en una cola de Doxiq) o external_callback (el cliente, en su sistema, vía webhook + callback). Los dos clientes actuales usan external (farmacia por WhatsApp, banco por POST /review); la cola interna queda lista para quien quiera que Doxiq hospede la revisión.
