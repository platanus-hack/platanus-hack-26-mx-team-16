01 Resumen ejecutivo

El veredicto en una pantalla

Todo lo que sigue está anclado a archivos reales del backend. Donde el sistema ya soporta algo, lo reusamos; donde no, lo proponemos.

✔ Lo que ya funciona
  • El motor Temporal es durable, con replay, reintentos e idempotencia por processing_job_id.
  • run_extraction_pipeline ya se extrajo como coroutine reusable — la semilla de "fase" ya existe.
  • Las reglas ya son plugin-style (WorkflowRuleKind) y el verdict aggregator está desacoplado.
  • Disciplina de blobs en S3 por referencia entre pasos (límite 2 MiB de Temporal).
✘ Lo que pelea contra este caso
  • Sí existen "dos tipos de workflow": WorkflowType.STANDARD vs ANALYSIS + 3 @workflow.defn legacy separados (el worker registra hoy 4: el intérprete PipelineInterpreterWorkflow ya existe en feat/re-arch).
  • La extracción no es invocable aislada: persist=False solo gatea checkpoints/webhooks, no detiene el cómputo.
  • Sin confianza por campo persistida; confidence_threshold está en spec pero sin cablear.
  • Las reglas no leen datos externos: knowledge_context llega [] y no hay Tool registry de lookup.
◆ Recomendación central
Colapsar los tipos de workflow en un único motor genérico que ejecuta PIPELINES versionados sobre una librería de fases de primera clase. Los dos puntos de integración (A · extract-only y B · process-case) se vuelven dos entradas al mismo grafo, no dos productos. El Human-in-the-Loop (HITL) por WhatsApp lo opera el cliente; Doxiq solo emite confianza + señales y recibe los datos validados.
02 El caso de uso

Farmacia → receta → cobertura

El ejemplo transversal de este documento: una farmacia fotografía una receta manuscrita y la envía por WhatsApp a un número operado por el sistema del cliente (no nuestro). Doxiq interviene en dos puntos del mismo pipeline — A · extract-only (foto → JSON + confianza por campo) y B · process-case (datos validados → verdicts de cobertura) — y expone una Tool que consulta el web service de pólizas del cliente. Si la confianza por campo es baja, Doxiq solo etiqueta needs_clarification: el lazo HITL por WhatsApp lo opera el cliente, y los datos confirmados re-entran por B sin re-correr OCR.

03 Diagnóstico de la arquitectura actual

Qué hay hoy, exactamente

Tres evidencias en el código real confirman que sí existen "dos tipos de workflow", y que las fases son detalle de implementación, no unidades reconfigurables.

El matiz que evita diagnosticar mal la causa raíz

⚠ Inyección de contexto: la abstracción existe, el flujo está roto
El protocolo del evaluador (EvalInputs en kind_protocol.py) ya declara knowledge_context. El problema es que se invoca con knowledge_context=[] hardcodeado (analysis_run_activities.py). No falta la abstracción: falta poblar el dato.
✔ Tools de lookup: la abstracción ya existe en feat/re-arch
El registro de Tools ya existe: ToolDefinitionORM → tool_definitions (uq tenant+name, transport, connection_account_id, schemas) + ToolCallSnapshotORM → tool_call_snapshots (status OK/DEGRADED), endpoints POST|GET /v1/tools y activity tool_lookup. Falta el update de Tools. ConnectionCapability ya incluye LOOKUP (junto a RECEIVE/SEND) y el enum de providers añade HTTP para Tools (backend/src/common/domain/enums/connections.py).

04 La propuesta

Arquitectura por fases componibles

Patrón: Pipes-and-Filters sobre un DAG (grafo dirigido acíclico) / pipeline tipado, interpretado por un único workflow genérico de Temporal — el motor, implementado como un intérprete genérico. Cada fase es un filter con contrato PhaseInput → PhaseOutput; la secuencia es dato versionado (un pipeline), no código.

◆ Tres mecanismos de "gate" distintos (no confundir)
1) confidence_gate — umbral por campo; solo etiqueta needsClarification, nunca falla el run (farmacia). 2) await_documents — espera de completitud del caso (capacidad de plataforma; ningún caso actual la usa como gate). 3) when: — predicado condicional a nivel de fase para saltarla (Circulares salta enrich si el juez se leyó bien). Comparten la idea de "decidir el siguiente paso" pero son fases/params distintos. La tercera pausa es human_review (aprobación humana; la introduce el caso Garantías y circulares la encadena ×2).

Ejemplo de pipeline — el caso farmacia

⚠ Nota — estado real del DSL
El registro de fases ya existe: PHASE_LIBRARY + register_phase + execute_pipeline (backend/src/workflows/application/pipelines/runtime.py), con 11 de 13 handlers registrados (incluido confidence_gate). Faltan evaluate_rules y synthesize: el análisis sigue corriendo por WorkflowAnalysisRunWorkflow, fuera del intérprete. Los pipelines son datos versionados: es lo que reemplaza las clases @workflow.defn por tipo.
A pipelines/pharmacy_extract_only.yaml
pipeline: pharmacy_extract_only
version: 3
entry: ingest
phases:
  - id: ingest
  - id: extract_text
  - id: classify_pages
  - id: extract_fields
    params: { doctype: "receta_manuscrita",
             emit_field_confidence: true }
  - id: confidence_gate
    params:
      thresholds: { medication: 0.85,
                    dose: 0.90, default: 0.75 }
      on_low: emit_needs_clarification  # no pausa
  - id: finalize
    params: { webhook_event: "document.extracted" }
B pipelines/pharmacy_process_case.yaml
pipeline: pharmacy_process_case
version: 3
entry: enrich   # re-entra tras validación
phases:
  - id: enrich
    tool: fetch_policy_data
    params: { snapshot: per_case,  # B1: sin caché — respuesta persistida por-caso (auditoría)
             timeout_ms: 8000,
             on_timeout: degraded }
  - id: evaluate_rules
    params: { rule_set: "coverage_rules",
             context: ["policy_snapshot"] }
  - id: synthesize
    params: { output_schema: "coverage_verdict_v2" }
  - id: finalize
    params: { webhook_event: "analysis_run.completed" }

Mapeo a Temporal

Fase = activity

El cuerpo del workflow (interpretar el pipeline) es determinista; el I/O vive en activities — igual que invoke_lambda hoy. Blobs por ref de S3 entre fases.

HITL = signal + wait_condition

await_clarification bloquea sin consumir cómputo, sobrevive caídas días, y se reanuda con el Signal que entrega el Endpoint B. Patrón "pause-on-token / resume-via-API" (≈ task tokens de AWS Step Functions: un token pausa el flujo hasta que la API lo devuelve).

Versionado determinista

Sellar pipeline_id + version en el input del run (reproduce el pipeline con la versión que inició). Pipelines inmutables append-only. Build IDs / Worker Versioning (mecanismo de Temporal) pinnea cada run a su Worker.

Provenance (estilo Dagster)

Cada output de fase = asset versionado (hash code_version + input_data_version) → linaje gratis y re-run por fase. Generaliza el re_extractor actual a cualquier fase.

05 Los dos puntos de integración

Dos endpoints, un mismo grafo

Auth por API key M2M tenant-scoped (no JWT). TenantApiKey ya existe (repos + service en backend/src/tenants/…), y los endpoints M2M están registrados: GET /v1/jobs/{job_id}, POST /v1/tasks/{task_id}/resolve, POST /v1/case (este último, stub declarado). pharmacy_id no se gestiona de nuestro lado: es un dato del body (mismo nivel que policyId) que solo se reenvía a la Tool de póliza — ni entidad, ni subtenant, ni dimensión de métrica. El único principal es el tenant (el agregador).

◆ Entrada y salida configurables (plan de Conexiones)
La entrada de archivos de A (y de todos los casos) es un Source configurable (webhook /v1/ingest/{token}, Drive, Email, WhatsApp) que alimenta ingest; la salida son Destinations (webhook/Drive/Slack) que consumen finalize. La entrada de datos estructurados (B POST /case, callbacks de HumanTask) queda como API aparte. → plan de Conexiones.

◆ El verdict es un JOIN (drug × policy), no un flag
Sigue el modelo FormularyItem de HL7 Da Vinci (recurso FHIR que modela la cobertura de un medicamento por plan): el mismo medicamento puede estar cubierto bajo una póliza y excluido bajo otra. Las razones son códigos enumerados (cf. los códigos estándar de rechazo farmacéutico NCPDP 70/75 y de elegibilidad X12 271 AAA), no prosa. Esquema de salida nuevo — hoy VerdictAggregator produce un resumen PASS/FAIL a nivel de run, no la matriz drug×policy.
06 El conector externo

La Tool de póliza

Caso completo: Recetas médicas →

"Dado policy_id + pharmacy_id, llamar al web service del cliente para traer formulario/cobertura" se modela como una Tool tipada, registrada y reutilizable, consumida por la fase enrich y referenciable desde reglas.

◆ Nivel: organization, no workflow (recomendado)
La ToolDefinition vive en el registro de la org — igual que ConnectionAccount (es tenant_id sin workflow_id) y DocumentType.is_shareable: se registra una vez y cada pipeline la referencia/bindea. El secreto es org-level (rotación, allowlist SSRF y auditoría en un solo sitio); el workflow solo declara qué tool usa, el mapeo de args desde @case y overrides no-secretos (timeout), más un scope de uso (qué workflows pueden invocarla). Por-workflow duplicaría secretos y perdería el reuso entre pipelines — es el mismo patrón "definir en la org, bindear en el workflow" que ya usan connections/ y los document types.

Registro estilo MCP (reusar/extender connections/)

La capacidad LOOKUP (3ª dirección: nosotros llamamos y esperamos respuesta síncrona) y el provider HTTP ya están en el enum (backend/src/common/domain/enums/connections.py). La duda "¿reusar ConnectionAccount o credenciales por-tool?" quedó resuelta en código: tool_definitions.connection_account_id reusa ConnectionAccount.

{
  "name": "fetch_policy_data",
  "connectionAccountId": "conn_clientX",
  "inputSchema":  { "policyId":"string", "pharmacyId":"string" },
  "outputSchema": { "formulary":"FormularyItem[]", ... },
  "transport": { "method":"POST", "auth":"oauth2|api_key",
                "timeoutMs": 8000 }
}

Timeouts · snapshot · idempotencia · fallos

  • Decisión B1: SIN caché — fetch en vivo por caso (el cliente lo pidió así). Igual se persiste un snapshot por-caso de la respuesta para auditoría/reproducibilidad (no como caché reusable). Como no hay caché, hace falta un path degraded claro si su servicio cae.
  • Timeout + fallback (<5-10s SLA): marcar degraded, no hard-fail.
  • Circuit breaker: abre tras umbral explícito; half-open probe para cerrar.
  • Infra nueva: el DEFAULT_RETRY_POLICY (≈2 intentos) no incluye breaker ni path degraded — requiere el wrapper de conector determinista + snapshot por-caso.
07 Confianza & Human-in-the-Loop

El confidence_gate decide; el cliente posee el lazo

Caso completo: Recetas médicas →

La fase extract_fields produce dos señales distintas (cf. ocr_score vs score de Veryfi, OCR comercial): legibilidad del manuscrito y certeza semántica del campo. Son fallos diferentes — ilegible → re-fotografiar; campo incierto → confirmar con el farmacéutico.

Por campo, no por documento

Persistir {medication:{value,confidence}} en WorkflowDocument para poder consultar "qué docs <0.8".

Umbral por costo del error

Modelo de activation conditions de AWS A2I + per-field thresholds de Azure (0.80 straight-through). El trigger es dato, no código.

Grounding por recuperación

Re-rankear nombres extraídos contra el formulario del cliente (la misma Tool): transcripción abierta → matching de conjunto cerrado (cf. RxLens, sistema académico de lectura de recetas: +19-40% Recall@3).

⚠ Captura de correcciones (active learning) — no existe hoy
Persistir cada corrección como (field, ocr_guess, human_value, confidence) por tenant construiría un léxico de manuscrito específico del cliente. Requiere una entidad/tabla nueva + el flujo de re-entrada que las registre. Trabajo nuevo, no un hook sobre algo existente.

11 Decisiones

Candidatas a ADR — estado

✔ El registro vivo está en decisiones.md
Todas las decisiones arquitectónicas están cerradas (A1–A7, más las 10 de negocio y las de conexiones) — ver decisiones.md. Solo B2 (identidad de fármaco) y B3 (contrato A/B/C) esperan confirmación del cliente; lo demás queda abajo como contexto histórico.

Pendientes de confirmación del cliente

5 ¿Identidad de fármaco? default provisional · B2
Default tomado: normalizar a canónico con una KB de fármacos provista por el cliente (mecanismo #kb / Tool); ejercita de paso el fix de inyección de KB (knowledge_context). Falta confirmar qué codificación maneja el cliente (código local, ATC, NDC, RxNorm) — checklist B2/B3 en decisiones.md.
9 ¿Validar el contrato A/B/C con el cliente real? default provisional · B3
Default tomado: avanzar con los shapes asumidos del doc — no bloquea M0–M4. Ningún referente público describe esta topología de cuatro partes (agregador + farmacia + WhatsApp del cliente + Doxiq): validar los shapes contra el contrato de integración real antes de congelar — checklist B2/B3 en decisiones.md.

Resueltas — contexto histórico

1 ¿DAG completo o secuencia lineal con ramas? resuelta · parcial
✅ Resuelta (parcial): secuencia lineal + predicado condicional when: por fase alcanza para los 3 casos (Circulares lo prueba: enrich corre solo when: juez_nombre.confidence < 0.80). DAG con paralelismo real diferido hasta que un caso lo exija.
2 ¿Dónde vive la definición de pipeline? resuelta · A1
✅ En DB como Pipeline/PipelineVersion (fuente de verdad en runtime), cargada y versionada por fixtures JSON (mecanismo existente command.py load/dump). El run sella pipeline_id + version al arrancar; versiones inmutables append-only (una edición = versión nueva). Contexto: se evaluó repo, DB o ambos — impactaba versionado y Worker Versioning.
3 ¿Sintaxis de Tool en reglas? resuelta · A2
✅ Ambos: fase enrich para el pre-fetch general (default, cubre los 3 casos) y token @tool.x(args) para lookups dirigidos desde una regla; los dos pasan por el mismo conector determinista con caché (la Tool nunca se llama como HTTP crudo desde el LLM). Se arranca con enrich; el token se suma cuando un caso lo pida. Convive con @slug, #kb, {{system}}.
4 ¿Caché de formulario: dónde y TTL? resuelta
✅ Resuelta — B1: sin caché; fetch en vivo + snapshot por-caso + path degraded (ver decisiones.md). Contexto histórico: se evaluó un snapshot por (policy_id, pharmacy_id, formulary_version) con TTL — aquí pharmacy_id era solo un componente de la clave (passthrough), no una entidad.
6 ¿Confianza: del LLM o calculada? resuelta · A6
✅ Por capas: (1) legibilidad bbox (ya existe), (2) confianza semántica vía auto-eval del LLM y/o consenso (analysis_consensus_samples ya existe), (3) cross-check contra referencia (formulario / API de jueces / reglas de formato) — la señal más fuerte. El confidence_gate combina; la plataforma de eval (A5) calibra umbrales.
7 ¿Esquemas versionados + eval harness? resuelta · A5
✅ Plataforma de eval completa (datasets gestionados, métricas, comparación de versiones, UI de revisión) como workstream propio — no bloquea M0–M6. El versionado de esquema viene gratis con pipelines/doctypes inmutables. La captura de correcciones humanas (active learning, A7) se integra a este workstream.
8 ¿Webhooks en ambos modos? resuelto · en curso
Resuelto (sí): se desacopla el dispatcher para que cualquier pipeline (incl. ANALYSIS/cobertura) emita el webhook analysis_run.completed (con el output_schema de cada caso como payload — W1). En curso: el camino de salida de ANALYSIS ya avanzó en feat/re-arch — analysis_run.completed está en WebhookEventType y existen summary_dispatcher.py + http_dispatcher.py (workflows/infrastructure/services/webhooks/); queda terminar el cableado del dispatcher desacoplado (F3). Ambos casos de banco dependen de esto.
