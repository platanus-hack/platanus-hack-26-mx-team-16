00 ¿Mapea? Veredicto

Es la prueba de que la re-arquitectura valía la pena

Un cliente de otra industria (banca) y un caso distinto (validación cruzada de 3 documentos + aprobación) entra en el mismo motor sin tocarlo: solo se escribe otro pipeline (otras fases en otro orden), otros document types y otras reglas. Cero clases de workflow nuevas.

✔ Encaja directo (reusa)
  • Multi-documento por caso (Anexo/Evaluación/Resolución) → WorkflowCase + DocumentType por tipo. Es el modelo base de hoy.
  • 1 archivo combinado o varios → la fase classify_pages ya separa N documentos por tipo dentro de un archivo.
  • Reglas de paridad cross-document → es literalmente el workflow ANALYSIS actual: reglas con tokens @anexo.campo vs @evaluacion.campo y scope ALL_DOCUMENTS.
  • Output ACEPTADO/RECHAZADO → verdict (PASS/FAIL) + output_schema custom + fase synthesize. Ya existe.
  • Carga por web service + espera de resultados → API machine-to-machine (M2M, key por tenant) + webhook saliente (lo que ya agrega la propuesta).
✦ Lo nuevo: la fase human_review
  • Análisis incremental (reuso, no fase nueva). El caso analiza al llegar ≥1 documento — no espera a tener todos los tipos, y admite varios del mismo tipo. Las reglas cross-document usan on_empty=SKIPPED (ya existe en el enum WorkflowRuleOnEmpty) → solo disparan cuando sus inputs están; el verdict se re-evalúa al llegar más. Sin barrera dura.
  • human_review — aprobación del banco. Doxiq emite review.pending con el borrador; un analista del banco revisa en su sistema y confirma con POST /review antes de emitir. Doxiq no construye la UI de revisión para este cliente — igual que en farmacia el Human-in-the-Loop (HITL) lo opera el cliente.
  • Las dos reusan el mismo mecanismo que await_clarification de farmacia: pausa durable con wait_condition + reanudar por signal de Temporal. No es infraestructura nueva.
◆ Diferencias respecto a farmacia
El banco no usa Tool externa (no hay lookup de póliza; el análisis es puro cruce entre documentos) → su pipeline omite la fase enrich. Su HITL lo opera el cliente (un analista del banco, vía callback review.pending → POST /review) — igual que en farmacia lo opera el cliente por WhatsApp. Doxiq no construye UI de revisión para este tenant. Mismo motor, pipeline distinto.
01 El caso

Del expediente a la resolución

El sistema del banco crea un caso, sube los documentos (juntos o separados), y espera la resolución. Doxiq junta los documentos, valida las reglas de paridad, (opcional) pasa por un analista, y devuelve ACEPTADO/RECHAZADO por webhook.

03 El pipeline del banco

Mismo motor, otra secuencia de fases

El pipeline del banco omite enrich y agrega human_review. El análisis es incremental (decisión B10; ver §00). Todo lo demás son las mismas fases de la librería.

YAML pipelines/bank_garantia.yaml
pipeline: bank_garantia
version: 1
entry: ingest          # se invoca por cada upload del caso
phases:
  - id: extract_text
  - id: classify_pages
  - id: extract_fields
  # sin await_documents: el análisis se re-ejecuta por cada upload
  # (≥1 doc basta; varios del mismo tipo OK)
  - id: evaluate_rules
    params: { rule_set: "garantia_paridad", scope: "ALL_DOCUMENTS",
             on_empty: "SKIPPED" }   # regla sin sus docs → SKIP, no falla
  - id: human_review
    params: { enabled: true, mode: "external_callback",
             emits: "review.pending", on_reject: "return_to_rules" }
  - id: synthesize
    params: { output_schema: "resolucion_v1" }  # ACEPTADO/RECHAZADO + motivos
  - id: finalize
    params: { webhook_event: "analysis_run.completed" }

on_reject: "return_to_rules" — si el analista rechaza, el run vuelve a evaluate_rules con la corrección aplicada; si aprueba, sigue a synthesize.

04 Reglas cross-document

La paridad entre documentos ya es expresable hoy

Cada regla compara campos de distintos tipos de documento usando los tokens @slug.path que ya existen. El scope ALL_DOCUMENTS le da a la regla acceso a todo el expediente.

◆ Esto es el ANALYSIS de hoy
El sistema de reglas ya soporta scope sobre todo el caso, tokens @slug.path, kind de regla plugin y agregación de verdicts (WorkflowAnalysisRunSummary). Para el banco se autoran reglas nuevas (un kind de "paridad" y uno de "firma"), no un motor nuevo.
05 La pausa durable

La revisión humana es una pausa durable

human_review pausa el workflow sin consumir cómputo (wait_condition de Temporal) y se reanuda con un signal. Es el mismo mecanismo que await_clarification (farmacia) y que await_documents (la espera de completitud, disponible en la plataforma para casos que la necesiten — garantías no la usa, va incremental). Solo cambia qué evento despierta la pausa.

07 Endpoints

Integración con el sistema del banco

Auth M2M (X-Api-Key tenant-scoped). El banco crea el caso, sube documentos (una o varias llamadas), y recibe la resolución por webhook. Idéntico patrón a farmacia, pero orientado a caso multi-documento.

POST /v1/cases  ·  POST …/documents
// 1) crear expediente
POST /v1/cases  { "pipeline": "bank_garantia",
                  "externalRef": "EXP-9981" }
  → { "caseId": "case_x", "status": "processing" }

// 2) subir docs (1 archivo con varios, o varios)
POST /v1/cases/case_x/documents   (multipart)
  { file: <pdf>, hint: "anexo" }   # hint opcional
  → { "jobId": "job_a", "detected": ["anexo"] }
// repetir para evaluacion / resolucion
GET /v1/cases/{id}  ·  webhook
// estado del caso (o SSE en /events)
GET /v1/cases/case_x
  → { "status": "in_review",        # o analyzing / resolved
      "receivedTypes": ["anexo","evaluacion","resolucion"] }
                  // informativo — no es un gate (B10)

// webhook final → el banco detona su acción
POST {bank_url}  "analysis_run.completed"
  { "caseId": "case_x", "externalRef": "EXP-9981",
    "resolution": "ACEPTADO",
    "ruleResults": [ {"rule":"R1_nombre","status":"PASS"}, … ],
    "reviewedBy": "analista@banco" }
◆ La aprobación la hace el banco (confirmado)
Doxiq emite el webhook review.pending con el borrador de resolución + resultados de reglas; el analista del banco revisa en su sistema y confirma con POST /v1/cases/{id}/review { decision: "APPROVED" | "REJECTED", reviewer }, que reanuda el pipeline (signal). Doxiq no construye la cola de revisión para este cliente (queda disponible para quien la quiera hospedar).

◆ Unificación útil
La ClarificationRequest (farmacia) y la ReviewTask (banco) son la misma idea: una tarea humana que pausa un pipeline. Conviene modelarlas como una entidad HumanTask con un kind (clarification | approval). Una tabla, dos usos. → Ver el diseño detallado de HumanTask y las fases de pausa. human_review/HumanTask se entrega en F6 del plan de implementación (≙ hito M5).
09 UI: revisión

La revisión vive en el sistema del banco

Para este cliente, Doxiq no construye la UI de revisión: entrega los datos en el webhook review.pending y el banco los muestra en su propio sistema. Abajo, una referencia de qué ve el analista (alimentado por Doxiq) y cómo confirma con POST /review.
