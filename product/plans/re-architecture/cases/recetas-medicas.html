00 Veredicto

El veredicto en una pantalla

Todo lo que sigue está anclado a archivos reales del backend. Donde el sistema ya soporta algo, lo reusamos; donde no, lo proponemos.

✔ Lo que ya funciona
  • El motor Temporal es durable, con replay, reintentos e idempotencia por processing_job_id.
  • run_extraction_pipeline ya se extrajo como coroutine reusable — la semilla de "fase" ya existe.
  • Las reglas ya son plugin-style (WorkflowRuleKind) y el verdict aggregator está desacoplado.
  • Disciplina de blobs en S3 por referencia entre pasos (límite 2 MiB de Temporal).
✘ Lo que pelea contra este caso
  • Sí existen "dos tipos de workflow": WorkflowType.STANDARD vs ANALYSIS. El worker registra hoy 4 workflows: DocumentSetProcessingWorkflow, DocumentSetFieldReExtractionWorkflow, WorkflowAnalysisRunWorkflow y el nuevo PipelineInterpreterWorkflow (el intérprete genérico ya existe en feat/re-arch; los 3 legacy siguen vivos).
  • La extracción no es invocable aislada: persist=False solo gatea checkpoints/webhooks, no detiene el cómputo.
  • Sin confianza por campo persistida; confidence_threshold está en spec pero sin cablear.
  • Las reglas no leen datos externos: knowledge_context llega []; el Tool registry de lookup ya existe en feat/re-arch, pero evaluate_rules aún no corre en el intérprete.
◆ Cómo entra en la re-arquitectura
Este caso se resuelve con un único motor genérico que ejecuta PIPELINES versionados sobre una librería de fases de primera clase. Los dos puntos de integración (A · extract-only y B · process-case) son dos entradas al mismo grafo, no dos productos. El lazo HITL lo opera el cliente (ver §07). → Propuesta de re-arquitectura · Detalle técnico.
01 El caso

Farmacia → receta → cobertura

Una farmacia fotografía una receta manuscrita y la envía por WhatsApp a un número operado por el sistema del cliente (no nuestro). Doxiq interviene en dos puntos y expone una Tool que consulta el web service de pólizas del cliente.

A · Extract-only

Subconjunto del pipeline: foto → JSON + confianza por campo; termina en finalize (el webhook es la salida).

HITL en el cliente

Doxiq emite needs_clarification; el lazo lo opera el cliente (ver §07).

B · Re-entrada

Los datos validados re-entran por B y evitan re-correr OCR. Pipeline re-entrante.

03 Los dos pipelines

Un grafo, dos entradas

A recorre ingest → extract_text → classify_pages → extract_fields → confidence_gate y termina en finalize, que emite document.extracted — el webhook ES la salida de A; B re-entra en enrich con los datos validados y corre evaluate_rules → synthesize → finalize. Cada pipeline es dato versionado, no una clase de workflow.

⚠ Estado de implementación
El registro de fases ya existe en feat/re-arch: PHASE_LIBRARY + register_phase + execute_pipeline (backend/src/workflows/application/pipelines/runtime.py), con 11 de 13 handlers registrados — incluido confidence_gate. Faltan evaluate_rules y synthesize: el análisis sigue corriendo por WorkflowAnalysisRunWorkflow, fuera del intérprete. Estos YAML son el DSL propuesto; los pipelines son datos versionados: es lo que reemplaza las clases @workflow.defn por tipo.
A pipelines/pharmacy_extract_only.yaml
pipeline: pharmacy_extract_only
version: 3
entry: ingest
phases:
  - id: ingest
  - id: extract_text
  - id: classify_pages
  - id: extract_fields
    params: { doctype: "receta_manuscrita",
             emit_field_confidence: true }
  - id: confidence_gate
    params:
      thresholds: { medication: 0.85,
                    dose: 0.90, default: 0.75 }
      on_low: emit_needs_clarification  # no pausa
  - id: finalize
    params: { webhook_event: "document.extracted" }
B pipelines/pharmacy_process_case.yaml
pipeline: pharmacy_process_case
version: 3
entry: enrich   # re-entra tras validación
phases:
  - id: enrich
    tool: fetch_policy_data
    params: { snapshot: per_case,  # B1: sin caché — snapshot por-caso (auditoría)
             timeout_ms: 8000,
             on_timeout: degraded }
  - id: evaluate_rules
    params: { rule_set: "coverage_rules",
             context: ["policy_snapshot"] }
  - id: synthesize
    params: { output_schema: "coverage_verdict_v2" }
  - id: finalize
    params: { webhook_event: "analysis_run.completed" }
◆ El gate de farmacia es uno de tres mecanismos (no confundir)
Farmacia usa confidence_gate: umbral por campo que solo etiqueta needsClarification, nunca falla el run. Los otros dos están previstos como capacidad de plataforma (mismo diseño): await_documents (espera de completitud — Garantías la evita yendo incremental) y el predicado when: por fase (Circulares salta enrich si el juez se leyó bien) — este último ya implementado en el intérprete (PhaseSpec.when). La pausa await_clarification es opcional (M5 ≙ F6 del plan) y reusa la primitiva de pausa durable — HumanTask(kind=clarification), ver detalle de HumanTask.
04 Endpoints A y B

Dos endpoints, un mismo grafo

Auth por API key M2M tenant-scoped (no JWT) — ya en feat/re-arch: TenantApiKey existe (repos + service en backend/src/tenants/…) y los endpoints M2M están registrados: GET /v1/jobs/{job_id}, POST /v1/tasks/{task_id}/resolve y POST /v1/case (este último, stub declarado). pharmacy_id es passthrough: dato del body que solo se reenvía a la Tool de póliza (ver §08).

◆ Entrada y salida configurables (plan de Conexiones)
La entrada de archivos de A (y de todos los casos) es un Source configurable (webhook /v1/ingest/{token}, Drive, Email, WhatsApp) que alimenta ingest; la salida son Destinations (webhook/Drive/Slack) que consumen finalize. La entrada de datos estructurados (B POST /case, callbacks de HumanTask) queda como API aparte. → plan de Conexiones.

◆ El verdict es un JOIN (drug × policy), no un flag
Sigue el modelo FormularyItem de HL7 Da Vinci (estándar de interoperabilidad de formularios de cobertura en salud): el mismo medicamento puede estar cubierto bajo una póliza y excluido bajo otra. Las razones son códigos enumerados, no prosa — como los códigos estándar de rechazo/elegibilidad farmacéutica (NCPDP 70/75, X12 271 AAA). Esquema de salida nuevo — hoy VerdictAggregator produce un resumen PASS/FAIL a nivel de run, no la matriz drug×policy.
05 La Tool de póliza

La Tool de póliza

"Dado policy_id + pharmacy_id, llamar al web service del cliente para traer formulario/cobertura" se modela como una Tool tipada, registrada y reutilizable, consumida por la fase enrich y referenciable desde reglas.

◆ Nivel: organization, no workflow (recomendado)
La ToolDefinition vive en el registro de la org — igual que ConnectionAccount (es tenant_id sin workflow_id) y DocumentType.is_shareable: se registra una vez y cada pipeline la referencia/bindea. El secreto es org-level (rotación, allowlist SSRF y auditoría en un solo sitio); el workflow solo declara qué tool usa, el mapeo de args desde @case y overrides no-secretos (timeout, reintentos), más un scope de uso (qué workflows pueden invocarla). Por-workflow duplicaría secretos y perdería el reuso entre pipelines — es el mismo patrón "definir en la org, bindear en el workflow" que ya usan connections/ y los document types.

Registro estilo MCP (reusar/extender connections/)

Capacidad LOOKUP (3ª dirección: nosotros llamamos y esperamos respuesta síncrona) — ya en el enum ConnectionCapability junto a RECEIVE/SEND, con provider HTTP para Tools (backend/src/common/domain/enums/connections.py). ✅ Resuelta (A3): el secreto vive en ConnectionAccount con capability LOOKUP; ToolDefinition guarda lo no-secreto.

{
  "name": "fetch_policy_data",
  "connectionAccountId": "conn_clientX",
  "inputSchema":  { "policyId":"string", "pharmacyId":"string" },
  "outputSchema": { "formulary":"FormularyItem[]", ... },
  "transport": { "method":"POST", "auth":"oauth2|api_key",
                "timeoutMs": 8000 }
}

Timeouts · caché · idempotencia · fallos

  • Decisión B1: SIN caché — fetch en vivo por caso (el cliente lo pidió así). Igual se persiste un snapshot por-caso de la respuesta para auditoría/reproducibilidad (no como caché reusable). Como no hay caché, hace falta un path degraded claro si su servicio cae.
  • Timeout + fallback (<5-10s SLA): servir snapshot stale y marcar degraded, no hard-fail.
  • Circuit breaker: abre tras umbral explícito; half-open probe para cerrar.
  • Infra nueva: el DEFAULT_RETRY_POLICY (≈2 intentos) no incluye TTL/serve-stale/breaker — requiere una capa de caché versionada + wrapper de conector.
06 Confianza & Human-in-the-Loop

El gate decide; el cliente posee el lazo

La fase extract_fields produce dos señales distintas (como el OCR comercial de Veryfi, que separa ocr_score de score): legibilidad del manuscrito y certeza semántica del campo. Son fallos diferentes — ilegible → re-fotografiar; campo incierto → confirmar con el farmacéutico.

Por campo, no por documento

Persistir {medication:{value,confidence}} en WorkflowDocument para poder consultar "qué docs <0.8".

Umbral por costo del error

El trigger es dato, no código — mismo patrón que las activation conditions de AWS A2I y los umbrales por campo de Azure Document Intelligence (0.80 para pasar sin revisión).

Grounding por recuperación

Anclar la salida a un catálogo cerrado: re-rankear los nombres extraídos contra el formulario del cliente (la misma Tool) — transcripción abierta → matching de conjunto cerrado (RxLens, lector académico de recetas, reporta +19–40% en Recall@3).

⚠ Captura de correcciones (active learning) — no existe hoy
Persistir cada corrección como (field, ocr_guess, human_value, confidence) por tenant construiría un léxico de manuscrito específico del cliente. Requiere una entidad/tabla nueva + el flujo de re-entrada que las registre. Trabajo nuevo, no un hook sobre algo existente.
07 UI: el lazo del cliente

El lazo vive en el WhatsApp del cliente

Para este caso, Doxiq no construye el bot ni la UI de clarificación: entrega la extracción con confianza por campo en el webhook document.extracted y el cliente decide cómo preguntarle al farmacéutico. 08 Decisiones

Resueltas y abiertas
