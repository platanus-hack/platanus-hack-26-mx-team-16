00 En simple + veredicto

Explicado como para un niño de 5 años

🧒 La historia
Llega una caja (el archivo TIFF) llena de cartas (los oficios). Cada carta la firma un juez y trae una lista de personas. Doxiq abre la caja, separa las cartas, y de cada carta copia la lista de personas y los datos del juez. Si la firma del juez está borrosa, Doxiq mira el nombre del juzgado (que se lee bien) y busca al juez en una guía de jueces. Después, dos personas revisan que todo esté bien: primero alguien de Doxiq, después alguien del banco. Recién ahí se entrega. 📦 → ✉️✉️ → 👥 → 🔎 → ✅✅ → 🏦

Técnicamente: este caso entra en la arquitectura sin tocar el motor. Es el primero que combina todo lo que ya diseñamos — la Tool externa (farmacia), la revisión humana (garantías) — y agrega lo único realmente nuevo: una interfaz de revisión (preview del documento + resultados editables), usada por dos audiencias en cadena.

✔ Ya resuelto por otros casos
  • Separar 1 archivo en N documentos (oficios) → fase classify_pages.
  • Extraer una lista (1 a cientos de personas por oficio) → extracción con schema de array.
  • Consultar el juez ilegible → fase enrich + Tool lookup_judge: idéntico a la Tool de póliza de farmacia.
  • Revisión humana → fase human_review + HumanTask: igual que garantías, pero dos veces.
  • Validaciones de formato (cédula, NIT, montos) → reglas de validación por campo.
✦ Lo nuevo de este caso
  • Interfaz de revisión (Doxiq la construye). Preview del documento + resultados editables. Es la UI que antes quedaba "disponible y sin usar" — aquí es producto, y para dos audiencias (analista Doxiq y analista banco).
  • Dos human_review encadenados con distinta audiencia → HumanTask suma una dimensión audience (doxiq_analyst | bank_analyst).
  • Enriquecimiento condicional: la Tool del juez se llama solo si la confianza del nombre es baja. Usa las ramas condicionales (when:) que ya existen en el intérprete (PipelineInterpreterWorkflow); este es el primer caso que las ejercita y confirma la decisión del DAG (cerrada como parcial: lineal + when:; paralelismo real diferido).
  • Escala: cientos de personas por oficio → muchas validaciones (fan-out: evaluación en paralelo acotado, a dimensionar).

03 El pipeline

El pipeline de circulares (usa casi todas las fases)

YAML pipelines/bank_circulares.yaml
pipeline: bank_circulares
phases:
  - id: ingest                 # TIFF
  - id: classify_pages         # 1 archivo → N oficios
  - id: extract_fields
    params: { doctype: "oficio", list_field: "personas_afectadas" }
  - id: validate_extraction
    params: { rule_set: "formato_persona" }   # cédula/NIT/monto
  - id: confidence_gate        # genérico: etiqueta needsClarification, nunca falla el run
  - id: enrich
    when: "juez_nombre.confidence < 0.80"   # el umbral vive en un único lugar: enrich.when
    tool: lookup_judge
    params: { key: "@oficio.juzgado", fill: "juez_nombre" }
  - id: human_review            # Revisión 1
    params: { audience: "doxiq_analyst", assignee: "internal_queue" }
  - id: human_review            # Revisión 2
    params: { audience: "bank_analyst", assignee: "internal_queue" }
  - id: synthesize
  - id: finalize
    params: { webhook_event: "analysis_run.completed" }

assignee: internal_queue = la cola la hospeda Doxiq (B4: ambas interfaces son nuestras); audience decide quién la ve.

04 TIFF → oficios → personas

Tres niveles anidados

El dato es anidado: un archivo tiene varias cartas, y cada carta tiene una lista de personas. La extracción produce esa estructura tal cual. El corte de oficios combina LLM por contenido + un Lambda dedicado de detección de sellos iniciales (B5).

05 El juez ilegible

Si el juez está borroso, preguntá por el juzgado

El sello tapa el nombre del juez (baja confianza), pero el nombre del juzgado se lee bien. La Tool lookup_judge consulta el API de jueces que Doxiq construye y mantiene (B7) usando el juzgado como llave. Es exactamente la Tool de póliza de farmacia, con otro nombre.

06 La doble revisión

Dos human_review en cadena, dos audiencias

Primero un analista de Doxiq afina; luego un analista del banco valida. Ambas son la misma fase human_review + HumanTask, encadenadas, con distinta audience. Las dos muestran preview + resultados.

◆ Diferencia con garantías
En garantías la revisión la hace el banco en su propio sistema (callback). Aquí Doxiq construye las dos interfaces (analista Doxiq y analista banco entran a Doxiq). Por eso assignee = internal_queue en ambas, con audience distinta — y por eso la UI de revisión deja de ser opcional.
07 Validaciones por campo

Reglas de formato, por cada persona

Cada persona de la lista pasa por validaciones de formato. Son reglas kind=format que ya encajan en el motor — corren por fila.

Cédula

Formato válido de identidad (longitud, dígito verificador) — @persona.cedula.

NIT (empresa)

Si el afectado es empresa, valida NIT en vez de cédula — regla condicional por tipo.

Montos

Numérico, no negativo, formato moneda — @persona.monto.

⚠ Escala (a dimensionar)
Un oficio con cientos de personas × varias reglas = muchas evaluaciones. El fan-out de evaluación existe (concurrencia acotada), pero hay que dimensionar: ¿validar por lote? ¿límite de personas por oficio? Es de las cosas a medir en la plataforma de eval (F11 del plan de implementación; decisión B6).

✔ La tesis, comprobada 3 veces
Tres industrias, tres pipelines distintos, un solo motor. Lo que cambia entre casos es: las fases que se incluyen, el orden, las reglas, los document types y los Tools. El motor es el mismo.

10 Decisiones — estado

Estado (ver decisiones.md)

1 · ¿Doxiq hospeda las 2 interfaces? ✅ B4

Resuelta (B4): Doxiq hospeda ambas interfaces — analista Doxiq y analista banco entran a Doxiq con login. Pendiente: detalle de acceso/roles del banco dentro de Doxiq.

2 · ¿Cómo se separan los oficios? ✅ B5

Resuelta (B5): el corte de oficios combina LLM por contenido + un Lambda dedicado de detección de sellos iniciales.

3 · Escala de personas ✅ B6

Resuelta (B6): validación por lotes + límite blando de personas por oficio.

4 · BD de jueces ✅ B7

Resuelta (B7): Doxiq construye y mantiene un API de jueces; la Tool lookup_judge consulta ese API (no el web service del cliente). Pendiente: estrategia de match del juzgado (exacto vs normalizado) y cómo se actualiza el directorio.

5 · Tipos de oficio ✅ B8

Resuelta (B8): un solo document type oficio con un campo tipo (congelar / descongelar / pedir información). Las reglas condicionan por ese campo.

6 · Ingesta diaria ✅ B9/W3

Resuelta (B9/W3): el banco sube cada TIFF por API M2M — un Source HTTP, POST /v1/ingest/{token}; el lote diario automático se agrega como otro Source (Drive/SFTP) en F12 — ver conexiones.
