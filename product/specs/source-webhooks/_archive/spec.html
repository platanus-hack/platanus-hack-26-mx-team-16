01 Objetivo

Permitir que sistemas externos empujen documentos a un workflow de Doxiq vía HTTP (estilo inbound webhook de n8n): cada workflow expone una URL de ingesta dedicada protegida por API key, capaz de recibir los documentos como binario (multipart), como URL pública descargable, o como base64, y un flujo presign de dos pasos para archivos muy grandes.

Al recibir archivos válidos, la plataforma:

  1. Sube cada archivo a S3 (streaming, sin cargarlo entero en RAM).
  2. Crea un workflow_document_set por archivo (el modelo existente es 1 set = 1 archivo).
  3. Despacha el pipeline Temporal existente (DocumentSetProcessingWorkflow: OCR → clasificación → persistencia de workflow_documents → extracción → validación).
  4. Responde 202 inmediato con los IDs y una status URL consultable.
  5. El resultado final puede llegar al cliente por sus destination webhooks ya existentes (document.extracted / document.failed).

2.2 Lo que FALTA (gaps que este spec cierra)

#Gap
🔴No existe NINGÚN endpoint público de escritura. Toda la API exige Authorization: Bearer + X-Tenant. workflows_router guarda con verify_workflow_access ⇒ la ingesta necesita un router público nuevo (/v1/ingest).
🔴No hay resolución de tenant sin header. El token de la URL + API key deben resolver tenant + workflow por sí solos.
🔴El upload actual carga TODO el archivo en RAM (await file.read()). Hace falta streaming a S3 (y presigned PUT para >100 MB).
🔴No existen modos {url} ni {base64}: no hay código que descargue una URL remota a S3 ni que decodifique base64.
🟡No hay API keys por workflow/tenant (solo ADMIN_API_KEY global y tenant.webhook_signature_key para firmar salida). La credencial inbound es nueva.
🟡ConnectionProvider.WEBHOOK solo tiene capability SEND; el grid de Orígenes no tiene tile webhook.
🟡WorkflowDocumentSetTrigger = USER/RETRY/ORPHAN_SWEEPER/SYSTEM — falta valor WEBHOOK.
🟡No hay rastro de auditoría de ingesta — se introduce source_webhook_deliveries.

3.2 Resueltas (decisión técnica del autor)

#Decisión técnica
9Router público dedicado /v1/ingest/{token}/…, registrado en config/main.py fuera de workflows_router. Sin JWT, sin X-Tenant: una dependencia resuelve token → SourceWebhook → tenant_id + workflow_id y valida X-Api-Key con hmac.compare_digest.
10Formato de credenciales: token URL = src_{secrets.token_urlsafe(24)} (token String(64) UNIQUE); API key = dxk_{secrets.token_urlsafe(32)} (api_key String(512)). Prefijos distintos de whsec_.
11Modo {url} se descarga inline con streaming a S3, timeout SOURCE_WEBHOOK_URL_FETCH_TIMEOUT (60 s/archivo), tope de bytes durante el stream y validación SSRF (https/http públicos; bloquear privadas, loopback, link-local, metadata).
12Streaming multipart: el endpoint NO hace await file.read(). FastAPI spoolea a disco las partes >1 MB; el use case sube con s3_client.upload_fileobj contando bytes — si supera el tope, aborta el multipart upload y rechaza con 413 (parcial).
13Presign: POST …/presign devuelve por archivo una presigned PUT URL (15 min) + un uploadToken firmado stateless (HMAC con JWT_SECRET_KEY). POST …/complete verifica firma + HEAD y recién entonces crea documents + sets. Tope SOURCE_WEBHOOK_MAX_PRESIGNED_FILE_SIZE (default 5 GB).
14Idempotencia a nivel delivery: header opcional Idempotency-Key (≤255). UNIQUE(source_webhook_id, idempotency_key); repetición ⇒ snapshot original (response_snapshot JSONB) con header Idempotent-Replay: true.
15Trazabilidad: nuevo valor WEBHOOK en WorkflowDocumentSetTrigger + columna source_webhook_id UUID NULL (FK SET NULL) en workflow_document_sets.
16Validación de contenido: además del whitelist, se validan magic bytes (%PDF, \x89PNG, \xFF\xD8\xFF). No se confía en el Content-Type declarado.
17Rate limit por webhook: key_func = "ingest:{token}", default SOURCE_WEBHOOK_RATE_LIMIT_PER_MINUTE = 60 (+ límite IP global). 429 con headers X-RateLimit-*.
18ingestUrl lo compone el backend (presenter) con settings.PUBLIC_INGEST_BASE_URL — el frontend nunca conoce el host del backend (regla BFF).
19Catálogo Orígenes: PROVIDER_CAPABILITIES[WEBHOOK] pasa a {SEND, RECEIVE}. No se crean ConnectionAccounts para source webhooks.
20Cierre del ciclo: la pareja natural es un destination webhook suscrito a document.extracted/document.failed; la correlación es files[].documentSetId == payload.documentSet.id.

3.3 Heredadas (no re-litigar)

  • Payload de eventos de salida, firma HMAC Svix, reintentos: product/specs/source-webhooks/standard-webhooks.md (cerrado 1–24).
  • Estructura Orígenes/Destinos, sidebar, permisos manage_connections vs admin de workflow: product/specs/connections/spec.md.
  • 1 archivo ⇒ 1 set ⇒ N workflow_documents clasificados: product/plans/sse-events/upload-document.md (D1–D7).

3.4 Abiertas (no bloqueantes)

  • Cifrado en reposo de api_key (y de los secrets existentes) — mejora transversal.
  • Quotas por tenant (archivos/día, GB/mes) — necesita modelo de billing.
  • Allowlist de IPs por webhook (params.allowedIps) — fase 2.
  • MIME types extra (docx, xlsx) — depende del pipeline.
  • Firma de respuesta del status endpoint — hoy basta con la API key.
  • Sweeper de deliveries RECEIVED colgadas (crash a mitad de request): job que las marque FAILED tras N minutos, liberando la Idempotency-Key (§5.9).

05 Superficie pública de ingesta

5.1 Resolución de identidad (dependencia nueva)

backend/src/common/infrastructure/dependencies/source_webhook.py:

async def get_ingest_context(
    token: str,                                  # path param
    api_key: str | None = Header(alias="X-Api-Key"),
    domain_context: DomainContextDep,
) -> IngestContext:                              # {source_webhook, tenant_id, workflow}
    webhook = await repo.find_by_token(token)    # índice UNIQUE
    if webhook is None: raise SourceWebhookNotFoundError          # 404
    if not api_key or not hmac.compare_digest(api_key, webhook.api_key):
        raise InvalidSourceWebhookApiKeyError                      # 401
    if not webhook.enabled: raise SourceWebhookDisabledError       # 403
    workflow = await workflow_repo.find_by_id(webhook.workflow_id, webhook.tenant_id)
    return IngestContext(webhook=webhook, workflow=workflow)
  • Sin JWT, sin X-Tenant: el token resuelve tenant/workflow (gap §2.2.2).
  • Comparación constante de la key. Igualación de timing obligatoria: cuando el token no existe, la implementación DEBE ejecutar igualmente hmac.compare_digest(api_key, DUMMY_KEY) antes de responder 404, para que un atacante no pueda enumerar tokens válidos midiendo latencia (404 rápido vs 401 lento).
  • Rate limit por token (§3.2.17) como segunda dependencia del router.

  • Descarga inline con httpx.AsyncClient.stream() → S3 upload_fileobj, timeout 60 s/archivo, corte si excede MAX_UPLOAD_SIZE.
  • SSRF guard obligatorio (extiende el validador de destinos, que hoy solo valida la URL inicial): solo http/https; resolver DNS y rechazar IPs privadas/loopback/link-local/metadata (169.254.169.254). Redirects manuales: follow_redirects=False y seguir cada Location a mano (máx 3) re-validando ANTES de cada hop. Anti DNS-rebinding: conectar contra la IP ya resuelta y validada, no re-resolver entre validación y conexión.
  • Fallo de descarga de UN archivo ⇒ ese item con error (delivery PARTIAL), no aborta los demás.

5.5 Modo base64 (JSON — integraciones low-code)

{
  "files": [
    { "fileName": "factura_001.pdf",
      "mime": "application/pdf",
      "contentBase64": "JVBERi0xLjcKJ..." }    // <= 25 MB decodificado

  • Límite menor (SOURCE_WEBHOOK_MAX_BASE64_FILE_SIZE, 25 MB): el JSON infla ~33% y el body se parsea en memoria. Decodificación incremental hacia el stream de S3.
  • Pensado para n8n/Zapier/Make y firmware que no puede hacer multipart.

5.6 Flujo presign (archivos muy grandes, hasta 5 GB)

  • uploadUrl: presigned PUT (nuevo FileRepository.get_presigned_put_url), expiración 15 min, Content-Type fijado.
  • uploadToken: HMAC stateless (§3.2.13) — sin tabla extra. En /complete la verificación es triple: (1) firma válida y no expirada; (2) webhookId del payload == webhook resuelto por el token de la ruta (un uploadToken emitido para otro webhook/tenant no sirve aquí); (3) HEAD del objeto: existe y tamaño ≤ declarado — si excede el tope se borra y se rechaza el item.
  • Validación de contenido en presign: como el PUT no pasa por la API, /complete hace un GET con Range: bytes=0-15 del objeto y valida los magic bytes (§3.2.16) antes de crear documents; mismatch ⇒ borrar objeto + item 415.
  • El secreto de firma debe ser persistente: JWT_SECRET_KEY (o el secreto dedicado) NO puede regenerarse en cada arranque — invalidaría los uploadTokens en vuelo. Rotación solo explícita por operador.
  • El S3 key sigue el patrón {tenant_id}/{file_id}/{safe_filename} con file_id generado en /presign.

5.9 Idempotencia (header Idempotency-Key)

Patrón delivery-first (normativo) La fila de source_webhook_deliveries se inserta al INICIO del request (status RECEIVED, con la Idempotency-Key si vino), antes de subir archivos o crear sets. Así el UNIQUE(source_webhook_id, idempotency_key) corta duplicados y carreras antes de que exista ningún efecto secundario (cero sets huérfanos). Al terminar, un UPDATE fija el status final + files + document_set_ids + response_snapshot.
Flujo POST con Idempotency-Key → INSERT delivery RECEIVED con UNIQUE(webhook, key). Insert OK: procesa archivos → sets, UPDATE status final + snapshot → 202 nuevo. Conflicto con delivery TERMINADA: devuelve response_snapshot con header Idempotent-Replay: true → 200 mismo cuerpo original. Conflicto con delivery aún RECEIVED (en vuelo): 409 ingest.IdempotencyInFlight (el cliente reintenta luego).
  • Sin el header, cada llamada crea sets nuevos (cada archivo recibe file_id uuid4 ⇒ processing_job_id distinto).
  • Si el proceso muere a mitad, la delivery queda RECEIVED (visible como incompleta en Receptions); los sets ya despachados siguen su curso normal y son visibles vía sets. Un retry del cliente con la misma key recibe 409 hasta que un operador o un sweeper futuro marque la delivery FAILED (TODO listado en §3.4).

06 Optimización para archivos grandes

CaminoTope/archivoMemoriaMecanismo
multipart100 MB (MAX_UPLOAD_SIZE)O(chunk)FastAPI spool ≥1 MB a disco → upload_fileobj multipart S3; conteo de bytes con aborto
{url}100 MBO(chunk)httpx stream → S3; Content-Length pre-check + corte durante stream
{base64}25 MB decodificadoO(body JSON)decode incremental → stream S3
presign5 GB0 en la APIPUT directo del cliente a S3; la API solo firma y verifica

Reglas:

  1. Prohibido await file.read() en el camino de ingesta (el use case nuevo StreamUploadFile reemplaza ese patrón; /documents/upload migra después — fuera de alcance).
  2. upload_fileobj con TransferConfig(multipart_threshold=8MB, multipart_chunksize=8MB).
  3. Mecanismo de corte: el conteo lo hace un counting reader que envuelve el file object y lanza FileTooLargeError al superar el tope; upload_fileobj propaga la excepción y boto3 aborta el multipart upload (si quedara basura, abort_multipart_upload explícito en el except). El item se reporta con error 413 item-level.
  4. boto3 es sync: cada subida corre con asyncio.to_thread (patrón ya usado por download_file_to_temp) — incluye el abort, que también es bloqueante. Nunca llamar boto3 directo en el event loop.

07 Integración con el pipeline

7.1 Despacho (reutilización del dispatcher)

Por cada archivo aceptado, el use case IngestSourceWebhookFiles invoca el WorkflowDocumentSetDispatcher existente:

  • El dispatcher no cambia su lógica: solo acepta y persiste source_webhook_id y el nuevo trigger. Idempotencia por processing_job_id intacta.
  • Workflow sin document types ⇒ comportamiento actual (warning + set con 0 documentos). La UI muestra advertencia preventiva (§9.3) — no se rechaza la ingesta.

7.2 STANDARD

  • caseId/caseName presentes ⇒ 400 ingest.CaseNotAllowed.
  • El set corre el pipeline y el resultado llega vía destination webhooks/status URL.

7.3 ANALYSIS (resolución de case)

  1. caseId (UUID): el case debe existir en ese workflow/tenant ⇒ si no, 400 ingest.CaseNotFound. Los N archivos van al MISMO case.
  2. caseName (string 1–255): find_by_name(workflow_id, name) —
    • 1 match exacto ⇒ se reutiliza;
    • 0 matches ⇒ se crea vía WorkflowCaseCreator (created_by=None, status DRAFT);
    • >1 matches ⇒ 409 ingest.AmbiguousCaseName.
    Requiere método nuevo WorkflowCaseRepository.find_by_name(workflow_id, tenant_id, name).
  3. Ninguno ⇒ ANALYSIS sin case ⇒ 400 ingest.CaseRequired.
Los webhooks de salida hoy son solo STANDARD; para ANALYSIS el cliente se entera por la API autenticada o el futuro spec analysis_run.completed. El status endpoint (§5.8) sí cubre ambos tipos.

7.4 Notificación de resultado (cierre del ciclo)

Dos vías Polling: GET /status/:deliveryId (status por set + documentos). Push (recomendado): destination webhook document.extracted / document.failed. Correlación: payload.documentSet.id == files[n].documentSetId.

11 Edge cases

  • Mezcla válidos/inválidos (mime/tamaño/descarga) ⇒ se aceptan los válidos, item-level error para el resto, delivery PARTIAL, HTTP 202.
  • Único archivo y excede el tope ⇒ 400 con files[0].error.code = storage.FileTooLarge (regla de §5.7: el HTTP global solo refleja el agregado); delivery REJECTED.
  • 0 archivos / files: [] ⇒ 400 ingest.NoFiles.
  • > 10 archivos ⇒ 400 ingest.TooManyFiles (validado ANTES de procesar ninguno — determinista; la delivery queda REJECTED con el motivo).
  • Mismo archivo dos veces sin Idempotency-Key ⇒ dos sets distintos (file_id nuevo). La solución es el header.
  • Idempotency-Key repetida con payload distinto ⇒ se devuelve el snapshot original (semántica Stripe).
  • Dos requests simultáneos con la misma Idempotency-Key ⇒ delivery-first (§5.9): el segundo INSERT choca con el UNIQUE antes de crear nada; si la primera sigue en vuelo ⇒ 409 ingest.IdempotencyInFlight; si terminó ⇒ replay del snapshot.
  • Crash a mitad de proceso ⇒ delivery queda RECEIVED (incompleta y visible); cero sets huérfanos invisibles — los despachados aparecen en la UI de sets.
  • caseName en STANDARD ⇒ 400 ingest.CaseNotAllowed. ANALYSIS sin case ⇒ 400 ingest.CaseRequired. Ambiguo ⇒ 409.
  • URL que redirige a IP privada ⇒ rechazo SSRF en cualquier hop (máx 3 redirects).
  • URL sin Content-Length ⇒ se permite; el corte ocurre durante el stream.
  • Content-Type mentiroso (dice pdf, magic bytes no) ⇒ 415 item-level.
  • Cliente corta la conexión a mitad del multipart ⇒ FastAPI aborta; abort_multipart_upload limpia S3; no se crea delivery.
  • Presign: PUT nunca llega / token expira ⇒ /complete item error ingest.UploadNotFound; huérfanos los limpia lifecycle rule S3 (24 h).
  • Presign: objeto más grande que lo declarado ⇒ HEAD lo detecta, borra objeto, item 413.
  • Presign: contenido no coincide con el MIME declarado ⇒ el GET Range: bytes=0-15 de /complete (§5.6) detecta magic bytes inválidos, borra el objeto, item 415.
  • Presign: uploadToken de otro webhook ⇒ /complete rechaza (payload webhookId ≠ webhook de la ruta), item ingest.UploadNotFound (404 uniforme, sin filtrar detalle).
  • Webhook deshabilitado a mitad de un flujo presign ⇒ /complete 403.
  • Workflow eliminado ⇒ FK CASCADE borra los source_webhooks; el token deja de resolver ⇒ 404.
  • Regeneración de api_key en carrera con ingesta en vuelo ⇒ la request en curso ya validó; las siguientes fallan 401.
  • Temporal caído al despachar ⇒ el dispatcher hace COMMIT del set antes de start_workflow; el ORPHAN_SWEEPER re-despacha sets PENDING huérfanos. El 202 sigue siendo veraz.
  • Tenancy: todo lookup filtra tenant_id derivado del webhook; un deliveryId o documentSetId ajeno ⇒ 404.

14 Criterios de aceptación

  1. Un workflow (STANDARD o ANALYSIS) puede crear ≥1 source webhook desde Connections ▸ Sources ▸ Webhook; al crear se muestran URL y API key una vez.
  2. POST /v1/ingest/{token} con X-Api-Key válida acepta multipart, {url} y {base64}; cada archivo válido crea su set (+ case en ANALYSIS) y dispara el pipeline; responde 202 con deliveryId, documentSetIds y statusUrl.
  3. Archivos > 100 MB ingresan por presign hasta 5 GB sin pasar por la RAM; ningún camino hace read() completo.
  4. Key/token inválidos ⇒ 401/404 sin filtrar info; disabled ⇒ 403; mime/tamaño inválidos ⇒ error item-level con delivery PARTIAL/REJECTED.
  5. Idempotency-Key repetida devuelve la respuesta original sin crear sets nuevos.
  6. La pestaña Receptions lista las recepciones con filtros y enlaza a los document sets; los sets muestran trigger WEBHOOK.
  7. La pestaña Setup genera ejemplos correctos y copiables en los 7 lenguajes para los 4 modos, con la ingestUrl real.
  8. El resultado llega por los destination webhooks y es correlacionable por documentSetId; el status endpoint refleja el estado vivo.
  9. Patrón BFF intacto: el frontend nunca llama al host del backend ni conoce PUBLIC_INGEST_BASE_URL por env (la URL viene del presenter).
