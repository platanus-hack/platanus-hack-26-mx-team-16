PanoramaDe qué se compone

Doxiq es una plataforma de extracción documental y análisis de reglas de negocio (OCR + LLM). En producción se reduce a tres tipos de pieza:

  • Contenedores de aplicación — nuestro código: el frontend, la API, dos workers y el panel de administración. No guardan estado; se pueden reiniciar y replicar libremente.
  • Servicios de plataforma y estado — el orquestador Temporal, la base de datos, la caché/cola y el almacenamiento de objetos. Aquí vive todo el estado; necesitan persistencia y respaldo.
  • Dependencias externas — SaaS y funciones fuera del clúster a las que llamamos por red: modelos LLM, extractores en Lambda, correo y WhatsApp, más el gestor de secretos.

Las cuatro piezas de aplicación de servidor comparten una misma imagen; lo que cambia es el proceso que arrancan. Eso simplifica el build y los despliegues: una imagen, varios roles.

Dos cosas a corregir al pasar a producción
  • Separar el worker de jobs de la API. Hoy el arranque de la API levanta también el worker SAQ dentro del mismo contenedor. En producción deben ser piezas independientes para poder escalar y fallar por separado.
  • Temporal con base de datos real. Hoy corre en modo desarrollo con almacenamiento embebido; no es apto para producción. Necesita respaldarse en la base de datos gestionada.

InventarioQué contenedores hay y para qué

Contenedores de aplicación nuestro código · sin estado

PiezaQué esPara qué sirve
web
Frontend		 Aplicación web Next.js (Node). Sirve la interfaz de usuario y actúa de BFF (backend-for-frontend): el navegador nunca habla directo con la API; todo el tráfico pasa por sus rutas de servidor, que reenvían a la API e inyectan las credenciales. Es la única pieza expuesta al navegador.
api
Backend API		 Servicio FastAPI (Python), escucha en el puerto 8200. Es el corazón del sistema: atiende todas las peticiones (desde el frontend, desde la API pública, y desde los canales de entrada como email / WhatsApp / webhooks), mantiene conexiones de streaming en vivo (SSE) hacia la UI y lanza los flujos de procesamiento en Temporal. No procesa documentos él mismo.
worker
Temporal Worker		 Proceso que ejecuta los workflows de procesamiento documental. Aquí ocurre el trabajo pesado: por cada documento/caso corre el pipeline (OCR, extracción con LLM, análisis de reglas, síntesis). Toma trabajo de la cola document-processing de Temporal. Es la pieza que más recursos consume.
jobs
SAQ Worker		 Worker de tareas en segundo plano sobre Redis. Ejecuta jobs asíncronos cortos y tareas programadas (p. ej. reintentos de entrega de eventos cada pocos minutos). Descarga a la API de trabajo que no debe bloquear una petición.
admin
Admin (Directus)		 Panel administrativo de terceros (Directus). Consola interna de operación y soporte. Usa su propia base de datos.

Plataforma & estado aquí vive el estado · respaldo + persistencia

PiezaQué esPara qué sirve
orq
Temporal Server		 Orquestador de workflows duraderos. Coordina el procesamiento: guarda el estado de cada flujo, reparte el trabajo a los Temporal Workers por colas, reintenta pasos fallidos y permite flujos que esperan (documentos pendientes, revisión humana). Necesita su propia base de datos.
db
PostgreSQL + pgvector		 Base de datos relacional gestionada, con la extensión de vectores. Almacén principal de datos del negocio (tenants, workflows, casos, documentos, resultados) y de los embeddings de la base de conocimiento (pgvector). También respalda a Temporal y al panel de administración. Es el estado compartido del sistema.
cache
Redis		 Almacén en memoria. Cola de trabajos del SAQ Worker y caché. Necesita persistencia porque la cola de jobs vive aquí.
blob
Object Storage (S3)		 Almacenamiento de objetos compatible con S3. Guarda los archivos: documentos subidos, adjuntos recibidos por los canales y resultados/exportaciones. Se accede por endpoint S3.

TopologíaCómo se relacionan entre sí

El navegador y los canales entran por el borde. El frontend reenvía a la API, que es el único punto que orquesta: escribe en Postgres, usa Redis y S3, y lanza workflows en Temporal. Los workers hacen el trabajo (los Temporal Workers toman tareas de Temporal y llaman a los servicios externos; el SAQ Worker consume su cola de Redis). Todo el estado aterriza en la capa de datos.

CargaQué esperamos que escale mucho

La carga no se reparte por igual. Dos piezas absorben los picos y deben poder crecer en número de réplicas de forma agresiva (las marcadas en caliente en el diagrama); el resto se mantiene casi fijo o crece despacio. La capa de estado no se replica como las piezas sin estado: se dimensiona en vertical y es el techo del sistema.

PiezaEscalaQué la hace crecer
Temporal Worker Mucho ↑↑ El que más escala. Cada documento es trabajo de segundos a minutos (OCR + LLM). En una ráfaga de ingesta la cola se llena y necesitamos muchas réplicas en paralelo; al vaciarse, bajan. Es la pieza a la que conviene dedicarle autoescalado agresivo.
Backend API Mucho ↑↑ Crece con usuarios concurrentes y con ráfagas de entrada por canales/API/webhooks. Además mantiene conexiones de streaming (SSE) abiertas, así que el límite suele ser el número de conexiones, no solo la CPU.
SAQ Worker Moderado ↑ Sube con el volumen de jobs y tareas programadas. Más liviano que el Temporal Worker; crece según el tamaño de su cola.
Frontend Moderado ↑ Acompaña a los usuarios concurrentes. Es barato de replicar; rara vez es el cuello de botella.
PostgreSQL Vertical Estado compartido: se dimensiona en vertical y, si hace falta, con réplicas de lectura. Es el techo del sistema: cuidar el número de conexiones, porque cada réplica de API y de worker abre conexiones contra él.
Redis Vertical Se dimensiona en vertical / alta disponibilidad. La cola de jobs vive aquí, no se reparte en muchas réplicas.
Temporal Server Estable Puede ir en alta disponibilidad, pero no «escala un montón»: su límite práctico es la base de datos que lo respalda.
Object Storage Gestionado Escala solo (es un servicio gestionado); solo crece el almacenamiento.
Directus (admin) 1 réplica Uso interno y esporádico; no necesita escalar.

Señales de escalado

Los workers escalan por cola, no por CPU

El Temporal Worker y el SAQ Worker deben crecer según el backlog (cuánto trabajo espera en su cola), no según la CPU. Así responden al instante a una ráfaga de ingesta y se apagan cuando la cola se vacía.

La API escala por tráfico y conexiones

Réplicas según peticiones y conexiones SSE abiertas. Al añadir réplicas de API y de worker, vigilar que el total de conexiones a Postgres no supere su límite — ese suele ser el primer muro.

Regla práctica Dimensionar primero Temporal Worker (techo alto, autoescalado por cola) y Backend API (tráfico + conexiones). Tratar Postgres como recurso escaso: el número de réplicas de cómputo está acotado por las conexiones que la base de datos aguanta.
