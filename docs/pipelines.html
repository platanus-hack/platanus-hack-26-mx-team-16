Doxiq · motor de pipelines

Pipelines, fases y políticas

Cómo Doxiq modela el procesamiento de documentos y casos: un workflow posee un pipeline versionado; el pipeline es una lista ordenada de fases; las políticas afinan las decisiones de cada fase; y el conjunto de capacidades visibles en la UI se deriva automáticamente de la receta vigente.

Workflow · 1:1 con Pipeline Pipeline · versionado, append-only Fases · 15 manejadores en PHASE_LIBRARY Políticas · 2 (completitud + activación) Capacidades · 10, derivadas de la receta

1Visión general

Doxiq corre cada subida de documento y cada avance de caso dentro de un workflow Temporal genérico cuyo comportamiento está descrito como datos: una receta inmutable de fases más dos políticas. No hay un workflow hardcodeado por caso de uso — todo se configura desde el editor.

📦 Workflow

Contenedor de tenant. Define qué se procesa (document types, knowledge) y posee 1:1 un pipeline.

🧪 Pipeline

Receta inmutable y versionada. Una nueva versión es append-only; los runs sellan pipeline_id + version.

🧩 Fases

Pasos ejecutables. Tienen kind (handler), config (parámetros) y un when opcional.

🎯 Capacidades

Etiquetas que el FE usa para gatear tabs/acciones. Se derivan de la receta — no se declaran.

📜 Políticas

Reglas que afinan el comportamiento de ciertas fases (umbrales, completitud, muestreo, severidades).

⚙️ Motor genérico

Un solo PipelineInterpreterWorkflow itera la receta. Cada kind resuelve a un handler registrado.

2Workflows

Un workflow es el contenedor que un cliente configura por tenant. Agrupa todo lo que necesita una línea de negocio: a qué document types aplica, qué knowledge base usa, qué eventos webhook le interesan, y — clave — un único pipeline que define cómo se procesa cada caso o documento.

ADR 0002 — 1:1 con pipeline. Un workflow posee exactamente un pipeline (workflow_id FK NOT NULL + UNIQUE). No hay pipelines compartidos entre workflows; si quieres reutilizar, duplica el workflow (deep-copy). Esto garantiza que editar un pipeline nunca afecte a otro workflow.

3Pipelines

Un pipeline es la receta que ejecuta el motor genérico. Tiene metadatos de identificación y un puntero (current_version) a la receta vigente. La receta en sí vive en una entidad aparte, PipelineVersion, donde cada publicación crea una fila nueva e inmutable.

Estructura de PipelineVersion

PipelineVersion
├── uuid              UUID de la versión (inmutable)
├── pipeline_id       FK al Pipeline
├── version           entero, único por pipeline
├── phases: list[PhaseSpec]
│     ├── id          estable, único dentro de la receta
│     ├── kind        PhaseKind (handler a invocar)
│     ├── config      dict de parámetros del handler
│     └── when        predicado opcional («corre sólo si …»)
├── output_schema     JSON-schema del payload final + webhook
├── activation_policy umbrales de confianza, sampling, stages L1/L2
├── completeness_policy required_types, auto_ready
└── created_at        sello temporal
¿Por qué inmutable? Cuando un run arranca, sella (pipeline_id, version) en su input. Aunque publiques una v+1 al día siguiente, el run en vuelo sigue ejecutando la receta vieja. Temporal usa ese sello para garantizar determinismo de replay (mismo input + misma versión ⇒ mismo resultado).

4Versionado e inmutabilidad

Publicar es siempre una operación append: nunca se sobreescribe una versión. Esto te da tres superpoderes prácticos:

  1. Seguridad en producción.

    Un cambio problemático no «rompe» runs ya en vuelo: cada uno sigue ejecutando la receta que selló al arrancar.

  2. Auditoría y rollback.

    El historial (tab Historial en el editor) lista todas las versiones. Restaurar es tan simple como pedir «vuelve a v(N)» — la receta original sigue ahí.

  3. Drafts sin tocar producción.

    El editor trabaja sobre un draft local. Al hacer Publicar el backend valida, sella v+1 y avanza current_version. Hasta entonces, lo que ven los runners es la v actual.

Validación al publicar. El endpoint POST /v1/workflows/{id}/pipeline/versions corre el mismo circuito de validación que el editor usa en dry-run. Errores típicos: pipeline.invalid_phases, pipeline.invalid_extractor, pipeline.invalid_policies — todos con código y detalle legibles, y un 422 estructurado.

5Fases (etapas)

Una fase es un paso ejecutable de la receta. Cada fase tiene un kind (handler registrado en PHASE_LIBRARY), un config con sus parámetros y, opcionalmente, un predicado when declarativo del tipo "juez_nombre.confidence < 0.7".

Scope: document vs case

Cada fase tiene un scope que indica sobre qué corre. Es importante porque la receta se ejecuta una vez por documento y una vez por caso, y el orden importa:

ScopeCuándo correEjemplos
document Una vez por cada archivo que recibe el caso. ingest, extract_text, classify_pages, extract_fields, assess, validate_extraction, finalize
case Una vez por caso, después de que todos los documentos están listos (o no, según la fase). await_documents, confidence_gate, await_clarification, enrich, analyze, human_review, output, deliver
Invariante de orden. La validación exige que todas las document-scope aparezcan antes de la primera case-scope. Por eso la fase await_documents (que es case-scope) suele ir inmediatamente después de la familia de extracción.

Catálogo completo de fases

Cada kind tiene un handler registrado y un configSchema que el editor consulta al endpoint GET /v1/pipelines/phase-catalog (el FE nunca debe hardcodear este catálogo).

El predicado when

Una fase puede llevar un when declarativo que el intérprete evalúa contra señales ya presentes en el run state. Si el predicado es falso, la fase se salta (no falla, no aborta). 

// Ejemplo: enriquecer sólo si la confianza del campo «curp» es baja
{
  "id": "enrich_curp",
  "kind": "enrich",
  "config": { "tool": "sat_curp_lookup", "on_failure": "review" },
  "when": "curp.confidence < 0.7"
}
Señales disponibles. _predicate_context expone facts numéricos del run: confianza por campo (fieldname.confidence) y cualquier cosa que una fase haya depositado en scratch.predicate_context. Un predicado sintácticamente válido cuya señal aún no existe sigue dejando correr la fase: es una condición de datos, no un error de configuración.

6Capacidades

Las capacidades son etiquetas booleanas que describen qué puede hacer un workflow. Lo importante: no se declaran — se derivan de la receta vigente mediante derive_capabilities(version). Esto elimina el clásico drift entre un flag en la entidad y las fases reales.

Mapa capacidad ↔ fase

CapacidadSe activa si…Aparece en la UI como…
extraction Hay alguna fase document-scope (la base). Siempre — es la base de todo pipeline.
multi_doc_dossier Existe await_documents. Tab Completitud en el editor + esperar docs en el caso.
analysis Existe analyze. Reglas de negocio y análisis ejecutándose en el caso.
layer2_confidence Existe assess. Score capa-2 visible en el detalle del campo.
enrichment Existe enrich. Tools firmadas disponibles en el editor y la cola.
clarification Existe await_clarification. Estado NEEDS_CLARIFICATION alcanzable.
human_review Existe human_review o activation_policy.stages está poblado. Colas de revisión L1/L2 en el módulo de staff.
structured_output Existen output y deliver. Salida estructurada + entrega a destinos.
fan_out classify_pages.config.fan_out == "child_cases". Child cases visibles en el detalle del caso padre.
qa activation_policy.qa_sample_rate > 0. Cola de auditoría QA para casos auto-aprobados.

Wizard «agregar capacidad»

Para no obligar al usuario a colocar manualmente cada fase + patch de policy, existe un wizard en el backend: POST /v1/workflows/{id}/pipeline/capabilities con { "capability": "analysis" }. Internamente:

  1. Lee la versión vigente

    Carga current_version del pipeline.

  2. Calcula addable_capabilities

    Capacidades registradas en ADDABLE_CAPABILITIES que aún no están presentes.

  3. Aplica la macro

    Inserta las fases necesarias en el orden canónico (document-scope → case-scope → await_documents primero) y, si la capacidad lo requiere, hace patch a activation_policy o completeness_policy. Idempotente: una capacidad ya presente ⇒ 409.

  4. Valida y publica v+1

    Mismo camino que el editor: valida la receta, sella una nueva PipelineVersion y avanza current_version.

El editor sigue siendo la vista avanzada. El wizard agrega el andamiaje correcto. Después puedes entrar a la pestaña Etapas y afinar el config, mover una fase, añadir un when, etc. — antes de publicar.

7Políticas

Las políticas son documentos JSON pequeños que se sellan junto a la receta. No son fases — son parámetros de las fases que leen ese JSON. Hay exactamente dos:

📋 ActivationPolicy

Gobierna confidence_gate y human_review. Umbrales de confianza, ruta clarify/review, muestreo, severidades bloqueantes y stages L1/L2.

📦 CompletenessPolicy

Gobierna await_documents. Cuántos documentos de cada tipo hacen al caso «listo» y si se debe marcar automáticamente como ready.

CompletenessPolicy

CompletenessPolicy = {
  "required_types": {            // slug del doc type → count ≥ 1
    "factura":        1,
    "identificacion": 1,
    "comprobante":    2          // p. ej. «2 recibos de nómina»
  },
  "auto_ready": false           // ¿cerrar expediente automáticamente al cumplir?
}
required_types
Mapa slug → count con cuenta mínima ≥ 1. El motor compara contra el snapshot de completitud (CaseCompleteness) del caso.
auto_ready
Si es true, en cuanto se cumpla el expediente el caso transiciona a listo sin intervención humana.

ActivationPolicy

Es la policy más rica. Gobierna tanto la compuerta de confianza (confidence_gate) como la posible revisión humana (human_review) y el muestreo. 

ActivationPolicy = {
  "field_thresholds": {           // "default" | "<campo>" | "<doctype>.<campo>" → [0, 1]
    "default":            0.75,
    "curp":               0.95,  // campo sensible: umbral alto
    "factura.total":      0.80
  },
  "on_low_confidence": "clarify", // "clarify" → pause & await_clarification
                                  // "review"  → directo a la cola humana
  "blocking_rule_severities": ["BLOCKER", "CRITICAL"],
  "sample_rate":         0.10,    // % de casos enviados a un humano ANTES de aprobar
  "qa_sample_rate":      0.05,    // % de casos AUTO-aprobados auditados DESPUÉS
  "mode":               "mandatory",
  "stages": [                    // revisión multinivel (opcional)
    { "stage": "review_l1", "mode": "mandatory" },
    { "stage": "review_l2", "mode": "by_exception" }
  ]
}
field_thresholds
Umbral por defecto y overrides por campo / por (doctype, campo). Clave "default" es el suelo común; cualquier otra clave con prefijo doctype. aplica solo a ese doc type.
on_low_confidence
Si "clarify", la fase confidence_gate enrutará a await_clarification. Si "review", directamente a la cola humana.
blocking_rule_severities
Severidades de reglas que, si aparecen, siempre bloquean (sin importar umbrales). Típicamente ["BLOCKER"] o ["BLOCKER","CRITICAL"].
sample_rate
Muestreo pre-aprobación: 0–1, porcentaje de casos que se enrutan a un humano antes de marcar como listo. 0 = todo auto-aprobado, 1 = todo va a revisión.
qa_sample_rate
Muestreo post-aprobación: 0–1, porcentaje de casos auto-aprobados que se mandan a una cola de auditoría QA después de COMPLETED. Activa la capacidad qa.
mode
Comportamiento del gate: "mandatory" (siempre) o "by_exception" (solo si hay señal).
stages
Lista ordenada de stages L1/L2. None ⇒ un solo gate (modo E4). Si está presente, debe ser review_l1 antes que review_l2 y no puede tener duplicados.

9Estados del caso

Una vez que la pipeline corre, lo que ve el usuario es la máquina de estados del caso. Cada estado es público (visible en la UI) y se transiciona por un único choke point: TransitionCaseStatus, que valida la legalidad, persiste y emite un case_events con status.changed.

10Eventos & webhooks

El pipeline emite eventos a un outbox. Los destinos (HTTP, S3, email…) se suscriben a eventos específicos de un workflow; el outbox garantiza entrega al menos una vez.

EventoCuándo se emiteLo emite
document.extractedUn documento terminó su pipeline de extracción y se persistió.finalize
document.failedUna fase de extracción falló de manera no recuperable.finalize (en error path)
case.splitclassify_pages con fan_out = child_cases repartió al menos un documento a child cases.classify_pages
case.output.readyEl caso terminó, se construyó y validó el output contra el output_schema.deliver
case.failedUna fase case-scope falló y el caso queda terminal.deliver (en error path)
children.completedEl último child de un padre entró a estado terminal ⇒ el padre cierra.maybe_complete_parent
Default subscribed_events. Si defines un output_schema pero no dices a qué eventos suscribirte, el backend propone ["document.extracted", "document.failed"] por compatibilidad con workflows STORAGE-only. Para entrega de casos completos, suscríbete explícitamente a case.output.ready.

11El editor del frontend

La página /workflows/<id>/pipeline es donde el usuario configura todo lo anterior. Tiene tres pestañas que se mapean 1:1 a los conceptos del modelo.

Qué hay en cada pestaña

PestañaQué muestraQué edita
Etapas La receta agrupada por scope (por documento / por caso), con cada fase como tarjeta que abre un sheet de config + when. Añadir/quitar/reordenar fases, editar su config, añadir un when, o usar el switch de capacidad para invocar el wizard.
Políticas Los dos bloques JSON: Completitud del expediente y Activación de revisión (umbrales, sampling, stages L1/L2, QA). Edita la policy in-place. El backend re-validará al publicar.
Historial Las versiones inmutables de este pipeline, con diff frente a la actual y opción de restaurar. Cambiar current_version a una previa (es lo más parecido a «rollback» en este modelo).

A. Glosario

Workflow
Contenedor de tenant. Posee 1:1 un Pipeline y un conjunto de capacidades derivadas.
Pipeline
Receta versionada. Tiene un current_version y un status (DRAFT/ACTIVE/ARCHIVED).
PipelineVersion
Fila inmutable. Contiene phases, activation_policy, completeness_policy y output_schema.
Fase / Phase
Paso de la receta. Tiene id, kind (handler), config y un when opcional.
Scope
document = corre por archivo; case = corre por caso.
Capacidad
Etiqueta booleana derivada de la receta: extraction, analysis, enrichment, human_review, qa, fan_out…
ActivationPolicy
Policy que gobierna la compuerta de confianza y la revisión humana (umbrales, sampling, severidades, stages L1/L2).
CompletenessPolicy
Policy que gobierna await_documents (cuántos docs de cada tipo, auto_ready).
output_schema
JSON-schema del payload final; valida case.output y arma el webhook.
Run
Instancia de Temporal que sella (pipeline_id, version) al arrancar.
Choke point
TransitionCaseStatus: única puerta para cambiar el estado del caso, con append de case_events.
Outbox
Cola de eventos con entrega al-menos-una-vez hacia destinos suscritos.

D. Reglas prácticas

  • Una capacidad no se declara, se induce. Si tu receta la habilita, la UI la muestra. Si no, la UI la oculta.
  • Editar nunca muta. Cambias el draft, validas, publicas v+1. Los runs en vuelo no se enteran.
  • Document-scope antes que case-scope, siempre. El validador lo exige; es lo que permite que el motor sea lineal-con-condicionales.
  • Capabilities ≠ fases sueltas. Algunas capacidades son 1 fase (assess ⇒ layer2_confidence), otras son 2 (output + deliver ⇒ structured_output) y otras son config (qa_sample_rate > 0 ⇒ qa).
  • Sample rate ≠ QA rate. Uno es antes, el otro es después. No los confundas.
  • El wizard es el camino feliz. Si tu flujo no requiere tunear a mano, usa «agregar capacidad» y deja que el backend coloque las fases en el orden correcto.
