Modelo de datos

Dónde Owliver guarda qué

Cuando Owliver escanea un sitio, todo lo que produce —el progreso, los hallazgos, los chatbots detectados, la calificación y los eventos en vivo— se guarda en una base de datos PostgreSQL. Aquí te mostramos qué tabla guarda qué cosa, cómo se relacionan y por qué están diseñadas así.

10 tablas del motor de pentest relacionadas por sites → scans → findings diseño anti-IDOR

Tres principios de base

Antes de las tablas, las reglas

Todo el modelo de datos se rige por tres ideas. Tenerlas claras hace que el resto encaje.

El cálculo lo hace código, no la IA

Todas las notas (web_score, overall_grade…) se calculan en Python antes de tocar la base de datos. La DB solo las guarda; así dos escaneos iguales dan el mismo resultado.

Cada hallazgo tiene identidad estable

Una "huella digital" (dedupe_key) identifica el mismo problema entre escaneos. Eso permite ver qué se arregló, qué sigue abierto y qué es nuevo a lo largo del tiempo.

El worker se observa desde la propia DB

Columnas como progress, tools_status y coverage dejan ver, en tiempo real, qué está haciendo un escaneo —sin montar herramientas extra de monitoreo.

Los contratos congelados

El formato exacto de un hallazgo

Antes de escribir nada en la base de datos, las herramientas devuelven sus resultados en un formato fijo de Python (Pydantic). Este "contrato" se congela en la hora 0 del proyecto y nadie lo cambia después: tocarlo rompería a varias personas a la vez. Es el lenguaje común entre los parsers, el agente de IA y el reporte.

Finding — un hallazgo

class Finding(BaseModel): source: Literal["owasp","agentic"] tool: str category: str # A01..A10 o LLM01..LLM10 title: str severity: Literal["critical","high", "medium","low","info"] cvss: float | None confidence: Literal["alta","media","baja"] description: str evidence: dict # payload, req/resp, screenshot affected_url: str | None endpoint: str | None param: str | None impact: str # lenguaje de negocio remediation: str # pasos accionables references: list[str]

AgenticResult — la superficie de IA

class AgenticResult(BaseModel): type: str # chatbot | prompt_input vendor: str | None # Intercom, Drift… o None location_url: str inferred_model: str | None # best-effort agentic_status: Literal[ "no_surface", "detected_not_tested", "tested"] findings: list[Finding] # source="agentic" # Los tres estados de agentic_status evitan # fingir que un sitio con chatbot no auditado # está limpio. Solo "tested" entra al promedio # ponderado del overall_score (60% web/40% IA).

Estos dos contratos viven en finding.py, uno de los 5 artefactos innegociables de la hora 0, junto con events.py (la forma de scan_events).

Decisiones de diseño

Por qué se guarda así (y dónde NO se guarda)

El modelo toma varias decisiones deliberadas. La mayoría existen para proteger datos sensibles —Owliver almacena vulnerabilidades reales y explotables— y para que el ranking y el monitoreo sean honestos.

scans.id es UUID, no un número

Un id secuencial (1, 2, 3…) dejaría adivinar y enumerar escaneos ajenos. Owliver guarda vulnerabilidades explotables de dominios privados: el UUID aleatorio cierra ese hueco (anti-IDOR).

El reporte público solo sale por token

Un escaneo privado nunca se expone por GET /scans/{id}. Para compartir se crea un token opaco con caducidad en public_reports, y sin los payloads de explotación.

Las capturas no se guardan en la DB

Las screenshots de evidencia se guardan como archivos en un volumen compartido y en la DB solo va una URL relativa. Ni base64 (inflaría la base) ni un servicio extra de almacenamiento.

penalty_raw se guarda sin tope

Cuando decenas de .gob.mx colapsan al mismo grado F, ¿quién va primero? La penalización cruda (sin el tope de 100) desempata el ranking "peores primero".

dedupe_key: la huella del hallazgo

Un sha256 determinista del sitio + tipo + URL + parámetro. El re-escaneo hace UPSERT por esa clave; lo que deja de aparecer pasa a fixed. Así se mide el progreso en el tiempo.

Idempotencia: no escanear dos veces

Un índice único parcial sobre scans(site_id, level) mientras está queued/running impide que un doble-clic lance dos ataques. El segundo intento devuelve el escaneo que ya existe.

Glosario

Términos de base de datos, en simple

Para leer las fichas de arriba sin tropezar con la jerga.

PostgreSQL

El sistema de base de datos donde vive todo. Guarda la información en tablas con filas y columnas, como hojas de cálculo relacionadas entre sí.

Tabla / fila / columna

Una tabla agrupa datos del mismo tipo (p. ej. scans). Cada fila es un registro; cada columna, un dato de ese registro.

PK (clave primaria)

La columna que identifica de forma única a cada fila (su "número de expediente"). En scans es un id UUID.

FK (clave foránea)

Una columna que apunta a la PK de otra tabla, creando la relación. findings.scan_id dice a qué escaneo pertenece el hallazgo.

UUIDv4

Un identificador largo y aleatorio (p. ej. a1f3…) imposible de adivinar. Se usa en vez de números seguidos por seguridad.

ENUM (enumerado)

Una columna que solo admite valores de una lista fija. status solo puede ser queued/running/partial/done/failed/cancelled.

jsonb

Un tipo de columna que guarda datos flexibles en formato JSON (estructuras anidadas). Ideal para evidence o coverage, que varían según el caso.

dedupe_key / sha256

Una "huella digital" fija que se calcula a partir de varios datos. Si dos hallazgos son el mismo problema, comparten huella, y no se duplican.

UPSERT

"Actualiza si existe, inserta si no". Permite re-escanear y actualizar los hallazgos ya conocidos en vez de crear copias.

Idempotencia

Que repetir la misma operación no cause efectos extra. Aquí: pedir el mismo escaneo dos veces no lanza dos escaneos.

Índice único parcial

Una regla que prohíbe filas duplicadas, pero solo bajo cierta condición (p. ej. solo mientras un escaneo está en curso).

IDOR

Fallo donde alguien accede a datos ajenos cambiando un id en la URL. El UUID + el control por dueño y por token lo previenen.
