Infraestructura

Dónde corre qué y cómo escala

Owliver es un conjunto de servicios en contenedores que viven en un servidor. Aquí explicamos qué servicio hace qué, cómo se aíslan los escáneres para no comprometer la seguridad, dónde se ejecuta cada herramienta y cómo debería crecer la plataforma a medida que llegan más usuarios y dominios.

Corre en un VPS Linux (no PaaS) 8 servicios en docker-compose el worker es el cuello de botella

El mapa físico

Cómo está montado todo

Todo vive en un único servidor (VPS) orquestado con docker-compose. El navegador habla con el frontend, este con la API, la API encola el trabajo en Redis, y el worker lo toma y ejecuta las herramientas. Lo clave: los escáneres salen a internet por una red separada y nunca pueden tocar la base de datos.

Red interna (postgres, redis) — sin salida a internet

Cómo se lanza cada herramienta

Dos formas de ejecutar, una decisión congelada

El worker ya es un contenedor; nunca lanza una herramienta "en el aire". Usa un patrón híbrido decidido de antemano: las herramientas ligeras corren dentro de su propia imagen, y las pesadas se lanzan como contenedores "hermanos" pidiéndoselo al Docker del servidor.

Ligeras → subprocess

Nuclei, testssl, Nikto, katana, ffuf, sqlmap, subfinder, dnsx… vienen preinstaladas en la imagen scanners. El worker simplemente las ejecuta como un comando más, sin Docker de por medio.

Rápidas, sin sockets ni permisos especiales.

Pesadas → run_tool() (DooD)

ZAP y hexstrike son contenedores enormes. El worker los lanza como "hermanos" montando el socket del servidor (/var/run/docker.sock) a través de un único helper que aplica red, límites de RAM/CPU y timeout.

Aislamiento real sin meter Docker dentro de Docker.
NO se usa DinD (Docker dentro de Docker). Exige modo --privileged, es lento y se rompe en la nube. Por eso los contenedores pesados son "hermanos" (DooD) y el montaje -v apunta al path del servidor, no del worker — el socket habla con el Docker del host, que solo conoce rutas del host.

Tiempos: cada herramienta tiene su límite

Un escaneo colgado bloquea la cola y arruina el live-view. Por eso hay un timeout duro por herramienta, un presupuesto global de ~8 minutos y un watchdog que aborta lo que falte. Si una herramienta falla, se anota y se continúa: nunca se pierden los hallazgos ya obtenidos.

Seguridad de la propia plataforma

Aislar los escáneres para no comprometerse a sí misma

Los escáneres atacan internet, así que son la parte más "peligrosa" dentro del servidor. Si no se aíslan, un atacante podría usarlos para llegar a la base de datos o a las credenciales del servidor (un SSRF lateral). El aislamiento se hace con redes de Docker separadas.

Red de egress para escáneres

Todo escáner se lanza en owliver_egress: puede salir a internet hacia los targets, pero no ve la red interna.

Red interna sin salida

Postgres y Redis viven en owliver_internal, sin egress y sin acceso desde la red de escáneres. La base de datos queda fuera de su alcance.

Metadata bloqueada

Se bloquean las IPs privadas y la de metadata del cloud (169.254.169.254), y el servidor del demo no monta credenciales: aunque un escáner la alcance, no hay secreto que robar.

Dos detalles que hacen que funcione

El live-view y el arranque en frío

Live-view: la verdad vive en Postgres

Redis pub/sub no recuerda lo que ya pasó: quien abre el stream tarde se perdería el inicio. Por eso cada evento se guarda en Postgres antes de publicarse, y el endpoint hace replay-then-tail: primero reproduce lo perdido desde la DB (ordenado por seq), luego sigue en vivo por Redis.

SSE con heartbeat de ~20s y compresión desactivada en esa ruta; auth por cookie (no por header).

Arranque en frío: todo pre-calentado

ZAP pesa ~2 GB y Nuclei descarga 12k+ plantillas en su primer uso —que falla por DNS dentro del contenedor—. Si eso pasa en vivo, el nivel básico no produciría hallazgos. Por eso, en el setup:

docker pull de todo con tags fijos (:stable, nunca :latest).
Plantillas de Nuclei pre-descargadas a un volumen + flag -duc en cada run.

Cómo debería escalar

El worker es el cuello de botella

Hoy todo corre en un solo servidor con max_jobs=1: un escaneo a la vez, porque cada uno consume varios GB de RAM (ZAP, garak). El frontend, la API, Redis y Postgres apenas se inmutan. Para crecer, lo que hay que multiplicar es el worker.

Escalado horizontal del worker

Como la cola vive en Redis (SAQ), basta con conectar más workers —en el mismo o en otros servidores— y todos toman trabajos de la misma cola.

Glosario

Términos de infraestructura, en simple

Contenedor / imagen

Una "caja" que empaqueta un programa con todo lo que necesita para correr. La imagen es la plantilla; el contenedor es una copia en ejecución.

docker-compose

Un archivo que describe varios contenedores y cómo se conectan, para levantarlos todos juntos con un comando.

VPS

Un servidor virtual que rentas (DigitalOcean, Hetzner…). Tienes control total: puedes montar el socket de Docker y salir libremente a internet.

PaaS gestionado

Plataforma que corre tu app por ti (más cómoda) pero con restricciones: normalmente no expone el socket de Docker, que Owliver necesita.

Socket de Docker

El "teléfono" para darle órdenes al Docker del servidor (/var/run/docker.sock). El worker lo usa para lanzar contenedores hermanos.

DooD vs DinD

DooD = Docker-fuera-de-Docker: el worker pide al Docker del host que lance contenedores "hermanos". DinD = Docker-dentro-de-Docker: más pesado y frágil; aquí se descarta.

subprocess

Ejecutar un comando como un proceso hijo del worker. Se usa para las herramientas ligeras que ya están en la imagen.

SAQ

La librería de cola de trabajos (sobre Redis) que reparte los escaneos a los workers de forma asíncrona.

Redis pub/sub

Mecanismo de mensajería en vivo: publica eventos a quien esté suscrito. No guarda historial, por eso la verdad vive en Postgres.

SSE

Server-Sent Events: una conexión por la que el servidor envía eventos al navegador en tiempo real. Alimenta el live-view.

Egress

El tráfico de salida hacia internet. Los escáneres lo necesitan; la base de datos no debe tenerlo.

SSRF lateral

Ataque donde se abusa de un componente con egress para alcanzar servicios internos o la metadata del cloud. Las redes separadas lo previenen.

Red bridge

Una red virtual de Docker que conecta contenedores. Owliver usa dos: una interna (sin internet) y una de egress (con internet).

Volumen

Almacenamiento persistente compartido entre contenedores. Aquí guarda la evidencia (capturas) en /data/scans.

Watchdog

Un vigilante que aborta el escaneo si se pasa del presupuesto de tiempo, para que una herramienta lenta no bloquee todo.

Escalado horizontal

Crecer añadiendo más máquinas iguales (más workers) en vez de una sola más grande. Es la vía principal de Owliver.
